## 二、新发现 P0：订单支付回调同款双重转换 bug

### 证据链（我已完整推演）

1. `backend/domain/order/router.py:249`（真实微信回调分支，`"resource" in parsed`）：
   ```python
   amount=Decimal(str(decrypted.amount)) / Decimal("100")
   ```
   `decrypted.amount` 已是网关层转好的"元"，此处再除 100 → 金额变为实际的 **1/100**
2. `backend/domain/order/service.py:227-230`：
   ```python
   if callback.amount != order.amount:
       raise PaymentError(f"支付金额不一致: 回调{callback.amount}, 订单{order.amount}")
   ```
3. **生产推演**：微信回调 120000 分 → 网关转 1200.00 元 → router 再转 12.00 → 校验 12.00 ≠ 1200.00 → **PaymentError**

### 定级：P0 生产阻断

- 影响面：生产环境**所有**真实支付回调（亲子课/观察期/会员/季度/半年）全部金额校验失败 → 用户付了钱，订单永远不变已支付，会员/押金全部无法开通
- 当前状态：未造成事故——MOCK_PAYMENT=true 阶段回调走 else 明文分支（`OrderPayCallback(**parsed)`）不触发；behalf/integration 测试均走 mock 明文分支，所以测试全绿但生产必炸
- **这是测试盲区型 bug**：只有真实网关加密分支能触发，mock 测试永远无法覆盖

### 全库排查（已确认无第三处）

```
grep -rn 'Decimal("100")' backend/ 结果：
- pay_v3.py:244      网关层 ✅ 正确
- mock.py:113        网关层 ✅ 正确
- order/router.py:249 路由层 ❌ 唯一残留 bug
```
