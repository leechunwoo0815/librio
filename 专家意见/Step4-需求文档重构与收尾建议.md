## 三、文档结构重构建议

### 3.1 当前结构问题

| 问题 | 实例 |
|------|------|
| 同一信息多处重复维护，必然漂移 | 会员状态机在 PRD 附录A、UML-ER 1.1、全流程 1.1、表结构 child 各写一份 → P0-1 |
| 全流程文档与 PRD 版本不同步 | 全流程文档残留 V2.0 书架模型 → P0-5 |
| 正式需求散落在非正式文档 | 管理后台操作规则只在"优化点拆解表"，其来源文档不在受控清单 → 维度一 P1 |
| 表结构文档自身注释与字段定义矛盾 | reservation 表字段含"已备"，自带状态流转注释却跳过"已备" → P0-4 |

### 3.2 唯一真源原则（Single Source of Truth）

| 信息类型 | 唯一真源 | 其他文档的处理方式 |
|---------|---------|------------------|
| 状态机 | UML-ER.md 状态图章节 | PRD 附录只放链接，不复制状态图；表结构枚举值必须与 UML 代码常量逐字一致 |
| 字段定义 | 表结构.md | UML-ER 的 ER 图只画关系，不重复字段类型 |
| 价格与规则参数 | 动态配置清单.md | PRD 正文引用配置键名（如"押金金额见 deposit_amount"），不写死数字 |
| 提示文案 | PRD 附录 G 文案规范 | BDD/全流程文档引用文案编号，不复述 |
| 权限矩阵 | PRD 附录 E | rbac.feature 只断言，不定义 |
| 业务流程叙述 | PRD 正文章节 | 全流程文档只画异常分支与状态联动，删除与 PRD 重复的正常流程叙述 |

## 四、需求粒度细化标准（V3.9 起强制执行）

每条业务规则必须满足以下模板，缺一不可：

```
【规则编号】DOMAIN-XXX（唯一、可追溯）
【规则描述】主语 + 触发条件 + 动作 + 结果，一句话可测试
【输入】字段名/类型/取值范围/来源
【计算】公式 + 取整规则 + 边界值（最大/最小/零/空/负）
【异常】每个失败分支的提示文案编号 + 状态去向
【配置】涉及的配置键名（无则写"写死，原因：..."）
【并发】多主体同时操作的裁决规则（无并发写"不适用"）
【验收】对应的 BDD 场景编号（无场景写"待补"，禁止空）
```

反例（当前 PRD 典型写法）："罚款金额 = 图书定价 × 1.5 倍"——缺取整、缺定价字段来源、缺余额不足异常、缺配置键名、缺验收场景，按新标准不合格。

## 六、终审结论

DmkWords V3.8 需求文档的**叙述层**（业务流程讲故事）基本完整，但**契约层**（状态机、枚举、公式、配置、权限）在四个文档间缺乏唯一真源管理，导致 6 个 P0 级矛盾直接进入开发。当前"177 单元测试 + 138 BDD 全绿"的测试报告不能作为需求稳定的证据——B-5/B-6 假绿与 @wip 跳过证明测试集本身已被污染。

建议冻结新功能开发，先按第二节路线图第 1、2 梯队完成文档修订（预计 3-5 个工作日），修订完成前任何涉及支付、退款、押金、状态机的代码变更都不应合入。
