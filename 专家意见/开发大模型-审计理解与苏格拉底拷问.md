## 第一部分：我对审计的总体理解

### 1.1 审计概况

专家审查组对 DmkWords V3.8 全套需求文档执行了 15 维度审计，覆盖 PRD、UML-ER、表结构、全流程文档、BDD 四个契约层 + 代码实现。产出 8 份报告，定性"不合格（需重大修订）"，共发现 **P0×6、P1×21、P2×25+**。

### 1.2 我认为审计最有价值的贡献

| 贡献 | 为什么重要 |
|------|-----------|
| **P0-6 iOS 虚拟支付合规零定义** | 这是真正的商业风险。小程序一旦拒审，iOS 端支付通道关闭=双端商业模式断裂。审计将它从 P1 升级为 P0 是正确的。 |
| **四方文档不一致（状态机/枚举/术语）** | 6 个 P0 中有 4 个是"各文档各说各话"导致的。如果只在 PRD 改而在表结构/BDD 不改，问题照样复现。"唯一真源"的治理方案是关键贡献。 |
| **BDD 假绿（B-5/B-6/RBAC @wip）** | "138 场景全绿"的测试报告确实有误导性。审计用实际 API 实测暴露了假绿，这是技术审计应有的实证精神。 |
| **退款公式边界条件补丁** | 取整规则、自然日口径、负数兜底、除数与会员天数一致——这些细节在需求文档中确实缺失，而代码必须处理它们。补丁非常务实。 |
| **Grill 12 条的纪律约束** | 特别认可：禁止盲信行号、禁止全局 sed、禁止"顺手优化"、禁止"把断言改弱让它过"。这些红线是开发者的行为守则。 |

### 1.3 我认为审计可以被挑战的几个方面

1. **审计的"文档完美主义"倾向**：6-phase plan 总计 12.5 工作日，其中阶段 0-3（纯文档）占比超过 60%。在一个正在迭代中的产品（26 领域模块、180+ API、239 测试全部通过）上，冻结一切代码变更两整周只改文档，在现实中是否可执行？

2. **补丁"拿来即用"假设**：审计报告中多次出现"补丁文本原样替换"。但我看了补丁文本——它们是**完美世界的楔子**。例子：附录 D 的"孩子切换时机限制"补丁说"听读进行中切换孩子，当前听读会话强制结束并结算时长"——这个机制涉及前端 WebSocket 连接、后端会话状态、数据库事务边界，至少需要 3-5 个代码文件修改。不能说这是"替换原文"就完事的。

3. **对技术债务的扁平化处理**：P0-5（书架容量矛盾）的问题根因是全流程文档 V2.0 残留。但代码到底用的是 V3.5 无上限模型还是 V2.0 20 本限制？如果代码已经实现了无上限，那这只是纯文档问题；如果代码仍保留 20 本上限，那就是真正的架构级 bug。审计报告未明确回答这个代码现状问题。

4. **测试修复的聚合力**：T5.1-T5.5（BDD 修复）共需要：修复 rbac.feature 环境依赖、补写 18 个新场景的 step 实现、修复月报 step 为端到端断言、术语统一改写、voice_reading 标记排除。专家估算 2 天。我补充一下增量：18 个新场景平均每个场景需要 3-5 个 step 定义 + 对应的 mock/seed 数据 + 环境上下文配置。保守估计 1 个场景 1 小时，仅此一项 = 18 小时。2 天（16 小时）覆盖全部 T5.x 可能偏紧。

## 第二部分：我的执行计划（对审计 6-phase 的调整提议）

### 总体原则

- **审计的 6-phase 结构 + 每个任务附 T 编号 + 审查验收清单继续保持**
- **但调整执行顺序和并行度**，优先保障资金安全和代码已存在的问题
- **区分"文档修改"和"代码修改"两个流**，允许它们按不同节奏推进

### 我提议的调整方案

#### 调整 1：阶段 0A（决策）+ 阶段 0B（代码现状调查）并行

审计的 T0.1（产品负责人答复 D1-D10）需要等待业务方。在等待期间，我不应空转。应同步执行 **0B：代码现状基线调查**：

```
0B-1: grep 押金枚举值（child.deposit_status / deposit_record.status）
       → 确认 REFUNDING=4 是否已存在于代码模型和数据库中
0B-2: grep 书架模型代码 → 确认 bookshelf_limit 实际实现是 0（无上限）还是 20
0B-3: grep 退款计算代码 → 确认 used_days 是自然日还是 24小时，取整方式
0B-4: grep 测验重考冷却 → 确认 cooldown=60min 代码位置
0B-5: grep 已关闭订单 → 确认迟到支付回调的当前处理逻辑
0B-6: grep BDD step 实现 → 确认 reading_stats 假绿的根因（mock 层恒真？404 没断言？）
0B-7: grep reservation 状态枚举 → 确认 "已备(PREPARED)" 是否在代码中存在
0B-8: grep book.price 字段 → 确认图书定价字段存在与否
```

产出：《代码现状基线调查报告》，作为所有 T 编号任务的输入。

**价值**：消除文档与代码之间"到底谁对谁错"的根本不确定性。没有这个基线，所有推断都是猜测。

#### 调整 2：阶段 1（资金安全）拆为文档流 + 代码流，同步推进

审计建议的顺序：Phase 1（文档）→ Phase 4（代码）。

我建议：**T1.2（退款公式）、T1.3（罚款超押金）、T1.5（退款拦截网）在代码和文档双端同步修改**。原因是：

- 这些任务是纯计算逻辑，状态机无关。文档定规则 → 代码调实现 → 新测试验证 → 三方同步。不阻塞其他任务。
- 反而如果先改文档等两周再改代码，第 14 天可能发现公式在代码实现中踩了坑（如前端金额展示精度、数据库 Decimal 精度、事务边界），又要回头改文档。
- **文档和代码的反馈环应该短到小时级，不是周级**。

#### 调整 3：阶段 5（测试修复）提前到与阶段 4 并行

审计在 Step 4 已注明"可与阶段 4 并行"。我进一步建议 **部分测试修复甚至提前到阶段 2/3**：

- T5.5（BDD 术语统一改写）可以在阶段 2（状态机术语统一）同时完成——同一个术语改动的两个落地面。
- T5.1（rbac @wip 修复）和 T5.3（voice_reading @future）是纯测试工程，与文档无关，可以开工头两天就做。
- T5.2(reading_stats 假绿)需要等阶段 4 确认 API 存在与否——但这不阻塞修复"step 实现绕过 API 层"的根因。

#### 调整 4：阶段排序优化

```
Week 1：
  M：T0.1（决策等待）∥ 0B（代码基线调查）
  T：T0.1 回复到达 → T1.2/1.3/1.5 文档+代码同步改（退款/罚款/拦截网）
  W：T1.1/1.6（iOS 回调）文档 + T2.4（押金枚举）文档+代码+T5.1（rbac @wip）
  T：T2.1/2.2/2.3（状态机统一）文档 + 对应代码+B+C+D+T5.3（voice @future）
  F：T2.5/2.6/2.7（术语+活动+书架）文档

Week 2：
  M：T3.1/3.2（多孩）文档 + T3.3/3.4（转让/冷却）文档+代码
  T：T3.5-3.12（规则补全）文档 + T4.2-4.8（代码对齐剩余部分）
  W：T5.2/5.4（BDD 修复+新场景）+ T4.1（押金枚举代码确认）
  T：T5.5（BDD 术语）+ 全量回归
  F：T6（回归+审查）
```

**总工时**：文档 3d + 代码 3d + 测试 2d + 审查 1d ≈ 9d（比审计估算少 3.5d 主要是通过并行压缩）

### 问题 1：关于"四方同步"的执行粒度

审计提出了"契约层修改必须 PRD、UML-ER、表结构、BDD 四方同步，缺一方不算完成"。这个原则好。但实际执行时，我发现**修改四方的时间差是不可避免的**：

```
Step: 改 PRD（2h）
→ 发现 PRD 修改后 UML-ER 状态图要跟着改
→ 改 UML-ER（1h）
→ 发现表结构枚举注释也要同步（0.5h）
→ 发现 BDD 场景断言要改（1h）
→ 发现还有全流程文档（不是四方之一，但也要改）
```

这实际上是一个链式传播，不是"四方同时改"。**专家是否认可链式修改的方法论？同步的标准是"最终全部修改完"还是"修改时必须四方一起提交"？**

### 问题 3：假绿修复的成本估算

T5.2（reading_stats 假绿）中提到修复方法"二选一：补真实 API 或删场景"。但让我拆解一下"补真实 API"的成本：

## 第五部分：我对标准补丁集的深度复核请求

以下是我对几个关键补丁的复核意见。审计说补丁可以"原样替换"，但我在实施前需要确认：

### 5.1 退款公式补丁（T1.2/批次2）

补丁说：used_days = (申请退款日期 - member_start_time 日期) 的自然日差，下限 1 天，上限总天数

**需要确认**：
- 如果申请退款日期 = member_start_time（同一日），自然日差 = 0，但下限 1 天。所以同日退款算 1 天？如果用户上午开通下午退款，被扣 1 天的钱，这合理吗？
- 闰年 2 月 29 日：如果 member_start_time = 2026-03-01，退款申请 = 2027-03-01，自然日差 = 365（2026-03-01 到 2027-03-01 是 365 天）。但 2024 年 3 月 1 日到 2025 年 3 月 1 日包含闰年 2 月 29 日但却是 365 天。这个口径需要白纸黑字确认。

### 5.2 罚款超押金补丁（T1.3/批次2）

补丁说"扣至 0 为止 + 差额挂 outstanding_fines + 押金不为负 + 挂账期间 5 项禁令"

**需要确认**：
- `outstanding_fines` 的累加规则：A 书丢书罚款 100 元（定价×1.5）- 押金余额 500 元 = 扣 100 后押金 400 元/超出部分 0 元。B 书丢书罚款 600 元 - 押金余额 400 元 = 扣 400（扣至 0） + 挂账 200 元。所以 outstanding_fines 应该分笔记录（一笔一笔的欠款）还是只记总额？分笔记录更利于家长逐笔理解/还款，但增加表结构和查询复杂度。
- 5 项禁令中的"禁止押金退款"和"禁止会员退款"是永久禁止还是 outstanding_fines 缴清后自动解除？

### 5.3 iOS 支付合规补丁（T1.1/批次5）

补丁说"ios_payment_mode（online / offline_guide）配置开关"

**需要确认**：
- 这个配置开关的读取时机：每次页面渲染时读，还是用户点击支付时读？如果是页面渲染时读，配置变更后 iOS 用户可能需要重新进入页面才能生效。
- Android 设备是否读取此配置？如果 Android 不管此配置始终显示在线支付，那后端需要根据 `user-agent` 或微信宿主环境判断来返回不同的支付参数。
- `offline_guide` 模式的支付流程：添加企业微信 → 咨询 → 线下转账 → 管理员后台创建订单。这个链路中家长付款后系统如何感知？是管理员手动标记到账，还是走对公账户自动对账？附录 J 说"与线上订单共用同一套会员开通事件"，但触发事件的角色是管理员还是系统？
