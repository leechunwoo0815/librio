## 二、BDD 与 PRD 直接矛盾清单（BDD 反向暴露的 PRD 缺口）

### 【B-1 | P1 | user_enrollment.feature：15-16 vs PRD：1.1 | 手机号"手动填写"与"微信授权自动获取"矛盾】

**原文引用：**
- BDD 场景大纲："并且 用户填写手机号为\"<手机号>\""（例子：13800138000）
- PRD 1.1 校验规则："手机号微信授权自动获取，不需要手动填写"

**问题分析：** PRD 明确手机号免填写（微信授权），BDD 却把"填写手机号"作为用户操作步骤。两者对同一流程的定义相反。若按 BDD 实现，前端要渲染手机号输入框；按 PRD 实现，不需要输入框。同时 PRD 未定义"微信授权拒绝/失败"的降级路径（是否回退到手动填写？），这才是 BDD 填写场景可能想覆盖的真实需求。

**建议补充的 PRD 文本（1.1 校验规则追加）：**
```
- 手机号优先通过微信授权自动获取；用户拒绝授权或授权失败时，
  降级为手动输入（11 位数字，前端格式校验），两种方式均可完成报名。
```

### 【B-2 | P1 | advancement.feature：66-71 vs 全流程文档：3.1 vs 集成测试实测 | 测验重考冷却时间三处矛盾】

**原文引用：**
- PRD 11.4："未通过：正确率 < 80% → 测验未通过…显示'可以重考'提示"（无冷却定义）
- 全流程文档 3.1："答对不到 4 题 → 未通过 → 20 分钟后可重考"
- 集成测试实测日志："quiz again → 1h cooldown — status=409"

**问题分析：** 同一规则三个版本：PRD 无冷却、全流程文档 20 分钟、代码实现 1 小时（409 冲突）。开发按哪个实现全凭运气；当前代码是 1 小时，与全流程文档的 20 分钟矛盾，且 cooldown 未进入动态配置清单（37 项中无 quiz_cooldown），属于写死的业务参数。

**建议补充的 PRD 文本（11.4 未通过追加）：**
```
- 重考冷却：测验未通过后须间隔 quiz_cooldown_minutes（默认 60 分钟，动态配置）方可重考；
  冷却期内发起测验返回 409 并提示"请 X 分钟后再试"。
- 同一本书测验通过后可再次测验（巩固练习），但不重复计分、不消耗冷却。
```

### 【B-4 | P2 | benefit_transfer.feature：19、29 | 转让剩余天数口径在 BDD 中固化但 PRD 未公式化】

**原文引用：** BDD："小红获得正式会员权益（剩余265天）"、"小红获得观察期权益（剩余20天）"。

**问题分析：** 365-100=265、30-10=20，BDD 隐含"总天数-已用天数"公式，但 PRD 3.1 只有示例无公式，且"已用天数"按自然日还是 24 小时制未定义（与维度四退款天数同源）。转让瞬间小数天如何取整（上午转让，当天算 0 还是 1）未定义。

**建议补充的 PRD 文本（3.1 权益继承追加）：**
```
- 剩余天数 = 原会员总天数 - 已使用天数（自然日口径，与 15.1 退款天数口径一致）
- 转让执行时刻的小数天向上取整计入"已使用天数"（宁多扣源孩子，不超给目标孩子）
```

### 【B-5 | P1 | reading_stats.feature：65-74 | 月报/分享场景 BDD 通过但 API 实测 404，用例假绿】

**原文引用：** BDD："场景： 查看月报…那么 显示月度阅读数据汇总"；"场景： 报告分享…那么 生成报告分享图片"。集成测试日志：`GET /report/stats/monthly` 404 × 4、`POST /report/share` 404 × 3，但 behave 执行显示通过。

**问题分析：** step 实现未真正断言 API 契约（可能在 mock 层恒真），导致"功能不存在但用例绿"。该问题已在维度八 P2 定性，此处补充根因要求：step 定义必须断言真实 HTTP 响应，不允许绕过 API 层直接断言数据库或 mock。

**建议动作：** 修复 steps 实现为端到端断言；将"月报 API"与"报告分享 API"补入 PRD 附录 C 端点索引（当前索引中 report 模块 8 个端点未列明细，无法核对）。

## 四、结论

1. **BDD 假绿是当前最大的质量风险**：月报/分享（B-5）、语音评测（B-6）、RBAC 整体 @wip 三类问题意味着"138 场景全绿"的含金量需打折扣，真实有效覆盖约 120 场景。
2. **术语漂移已渗透进 BDD 断言层**（B-3），单纯改文档不改 feature 文件无法根治。
3. **18 个 PRD 规则零用例覆盖**，其中季度/半年会员、会员升级、缓冲期续费是 V3.8 新上线的资金功能，属必须补齐项。
4. 建议按第三节表格的顺序补写 BDD（前 6 项为本迭代强制），同时按第二节的建议文本回填 PRD。
