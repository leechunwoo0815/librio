## 二、Top 3 核心矛盾深度拆解

### P0-1：会员"已关停"状态在三份文档中定义矛盾

**矛盾双方：**

| 文档 | 位置 | 原文 | 定义 |
|------|------|------|------|
| 全流程文档 | 1.1 会员状态流转 | "缓冲期 → 超15天 → 已关停" | 存在"已关停"状态 |
| PRD 附录 A | 会员状态机 | "体验(0) → 观察期(1) → 正式(2) → 已过期(3) → 已退出(4)" | 没有"已关停"状态 |
| UML-ER | 1.1 会员状态流转图 | "正式会员 → 已过期 → 续费成功（缓冲期内9折）" | 没有"已关停"状态 |
| 表结构 | child.status | "0=体验用户 1=观察期 2=正式会员 3=已过期 4=已退出" | 没有"已关停"状态 |

**导致的开发偏差：**
- 后端实现可能添加了一个"已关停"状态（如 status=5），但 PRD 和数据库设计没有预留
- 或者后端按 PRD 实现（无已关停），但运营流程要求有"已关停"概念
- 定时任务 check_observation_expiry 到期后究竟转到哪个状态？EXPIRED 还是已关停？

**建议的正确定义：**
要么在 PRD 附录 A、UML-ER 1.1、表结构 child.status 中统一添加"已关停(5)"状态，要么从全流程文档中删除"已关停"概念，统一为"已过期"后长期未续费即视为流失。

---

### P0-2：订单状态机在 PRD、UML-ER、表结构三者间矛盾

**矛盾双方：**

| 文档 | 位置 | 状态定义 |
|------|------|----------|
| PRD 第 5 章（审计基准） | 订单状态机 | 待支付(0) → {已支付(1) / 已关闭(5)}；已支付(1) → {已完成(2) / 退款中(3)}；退款中(3) → 已退款(4) |
| UML-ER 1.2 | 订单状态流转图 | 待支付 → 已支付 → 退款中 → 已退款；退款中 → 退款失败；退款失败 → 退款中 |
| 表结构 order.pay_status | 9.1 订单表 | 0=待支付 1=已支付 2=支付失败 3=退款中 4=已退款 5=已关闭 |

**矛盾点：**
1. 审计基准定义了"已完成(2)"状态，但 UML-ER 和表结构都没有
2. UML-ER 定义了"退款失败"状态，但审计基准和表结构都没有
3. 表结构定义了"2=支付失败"，但审计基准和 UML-ER 都没有

**导致的开发偏差：**
- 支付失败后订单状态应该是什么？审计基准没有定义
- 退款失败后如何重试？UML-ER 有定义但审计基准没有
- "已完成"和"已支付"的区别是什么？只有审计基准有定义

**建议的正确定义：**
统一为：待支付(0) → {已支付(1) / 已关闭(5)}；已支付(1) → 退款中(3)；退款中(3) → {已退款(4) / 退款失败(6)}；退款失败(6) → 退款中(3)。删除"已完成(2)"状态，或在所有文档中统一添加。

### P0-3：押金状态枚举在 child 和 deposit_record 表中不一致

**矛盾双方：**

| 表/文档 | 字段 | 枚举定义 |
|---------|------|----------|
| 表结构 child | deposit_status | 0=未交 1=已交 2=已退 3=已扣 |
| 表结构 deposit_record | status | 0=未付 1=已付 2=已退 3=已扣 |
| PRD 7.4 | 押金状态 | 未缴(0) → 已缴(1) → 退款中(4) → 已退(2) / 已扣(3) |
| UML-ER 1.7 | 押金记录状态流转图 | 未付(0) → 已付(1) → 退款中(4) → 已退(2) / 已扣(3) |

**矛盾点：**
child.deposit_status 和 deposit_record.status 都没有定义 4=退款中，但 PRD 和 UML-ER 都定义了。

**导致的开发偏差：**
- 当孩子申请押金退款时，child.deposit_status 应该更新为什么值？表结构没有 4=退款中
- 后端可能强行使用 child.deposit_status=4，但数据库枚举不允许
- 或者后端不更新 child.deposit_status，导致前端显示状态不一致

**建议的正确定义：**
在 child.deposit_status 和 deposit_record.status 中统一添加 4=退款中。

## 三、其他 P0 级问题

### P0-4：预约状态机不一致

**矛盾点：**
- PRD 第 8 章：只定义了 PENDING(0) / FULFILLED / EXPIRED
- UML-ER 1.8：定义了 待取(0) → 已备(1) → 已取(2) / 取消(3)
- 表结构 reservation.status：0=待取 1=已备 2=已取 3=取消
- BDD reservation.feature：用"RESERVED"和"PICKED_UP"

**问题：**"已备"状态在 PRD 中完全没有定义，BDD 术语与 PRD 不一致。

---

### P0-5：书架容量定义矛盾（V3.5 核心变更未同步到全流程文档）

**矛盾双方：**

| 文档 | 位置 | 原文 |
|------|------|------|
| PRD 5.2 | 加入书架 | "不限数量：即使书架已有 50 本，仍可继续添加（V3.5 取消 20 本限制）" |
| 动态配置清单 八 | bookshelf_limit | "0 = 书架最大数量，0表示无限制" |
| 全流程文档 4.1 | 书架借阅 | "图书馆浏览 → 加入书架（最多20本）" |
| 全流程文档 4.2 | 异常处理 | "书架满了（由 bookshelf_limit 配置决定，默认20本）→ 提示'请先完成当前阅读或手动移除'" |

**矛盾点：**
PRD 和动态配置清单明确书架无上限（V3.5 已取消 20 本限制），但全流程文档仍按旧版 V2.0 描述"最多 20 本"且定义了"书架满了"的异常处理。这是 V3.5 核心变更未同步到全流程文档。

**附带问题：**
全流程文档 10.1（借阅异常表）写"借阅时已达上限（由 bookshelf_limit 配置决定，默认20本）"，把 bookshelf_limit 和 borrow_limit 两个不同概念混用。借阅上限应由 borrow_limit 控制，不是 bookshelf_limit。

## 四、P1 级严重模糊（共 8 个）

### P1-1：晋级所需测验数定义不一致
- BDD advancement.feature："已通过5次测验"（固定值）
- PRD 11.5："quiz_pass_count 可配为 1-5"
- 动态配置清单："quiz_pass_count 默认=required_books"

### P1-2：default_required_books 已废弃但 PRD 还在用
- 动态配置清单："default_required_books 已废弃，代码读取 Level.required_books 字段"
- PRD 11.1："每级默认需读完 5 本书"

### P1-3：借阅状态术语不统一
- BDD：用"BORROWED"
- PRD：用"BORROWING"
- 表结构：用"借出"

### P1-4：预约状态术语不统一
- BDD：用"RESERVED"和"PICKED_UP"
- PRD：用"PENDING"和"FULFILLED"
- 表结构：用"待取"和"已取"

### P1-5：观察期评估结果与正式会员报名前置条件模糊
- PRD 1.3："前置条件：孩子观察期评估结果为'通过'"
- observation_evaluation 表有 result=1(通过)/2(待定)/3(未通过)
- "待定"状态下能否报名正式会员？未定义

### P1-6：活动报名收费活动状态流转不完整
- PRD 2.2：收费活动"报名状态为'待审核'（等待支付）"
- activity_enrollment.status 的 0=待审核 与订单支付状态如何关联？未定义

### P1-7：书架与借阅的关系未完全解耦
- PRD 5.1："书架与借阅完全分离"
- BDD bookshelf.feature 有"从书架预约借阅实体书"场景
- PRD 5.5 也有"从书架预约借书"流程，与"完全分离"表述矛盾

### P1-8：iOS 虚拟支付合规风险未定义
- PRD 全文未提及 iOS 端虚拟支付规避方案
- 维度十三要求审计 iOS 红线，但 PRD 完全没有定义
