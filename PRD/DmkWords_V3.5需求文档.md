## 一、用户报名流程（user_enrollment.feature）

### 1.1 亲子课程报名（99 元）

**业务背景**：亲子课是 DmkWords 的获客入口。家长支付 99 元，带孩子到线下门店参加一次 75 分钟的英文阅读体验课。体验课结束后，老师会出具一份孩子的英文阅读能力评估报告，作为后续报名观察期的前置条件。

**报名流程**：

1. 家长在小程序首页选择场馆（人广馆/徐汇馆等）
2. 选择上课时间（周六/周日固定时段，每场限 30 人）
3. 填写信息：家长姓名、手机号（微信授权自动获取）、孩子中文名、英文名、年龄（3-15 岁）、年级
4. 点击"立即支付 99 元"→ 调用微信支付 V3
5. 支付成功 → 生成订单（type=1, pay_status=1）→ 生成电子凭证

**校验规则**：
- 所有字段必填，否则支付按钮置灰
- 所选时间段名额已满 → 提示"该时间段名额已满，请选择其他时间"
- 手机号微信授权自动获取，不需要手动填写

**数据流转**：支付成功 → 创建 Order（type=PARENT_COURSE）→ 发布 order.paid 事件

### 1.2 观察期报名（500 元/30 天）

**业务背景**：亲子课结束后，如果家长认可教学质量，可以为孩子报名观察期。观察期是一个 30 天的完整体验期，孩子享受与正式会员**完全相同**的服务（无任何功能限制），包括：无限量阅读所有图书、音频伴读、查词、测评、打卡、排行榜。观察期结束时系统自动生成一份详细的阅读报告，引导家长转为正式会员。

**报名流程**：

1. 前置条件：孩子已完成亲子课程并获得测评报告
2. 家长选择观察期开始日期（默认当天）
3. 点击"立即支付 500 元"→ 调用微信支付
4. 支付成功 → 生成订单（type=OBSERVATION）→ 发布 order.paid 事件
5. 事件处理器自动：孩子状态变为观察期（status=1）、分配一对一指导老师（teacher_id）

**关键规则**：
- 观察期用户享受与正式会员**完全相同**的服务，无任何功能限制
- 观察期固定 30 天，从支付当天开始计算
- 到期前 7/5/3/2/1/当天 系统自动发送续费提醒
- 到期后自动生成观察期报告（阅读本数、词数、测验通过数、老师评语）
- 到期后进入 15 天缓冲期，续费可享 9 折

**前置条件校验**：
- 孩子未完成亲子课程 → 提示"请先完成亲子课程并获得测评报告"，无法进入报名页

### 1.3 正式会员报名（5400 元/年）

**业务背景**：观察期结束后，家长可以选择为孩子报名正式会员。正式会员有效期 365 天，享受全部服务。同一用户第 2 个孩子起享 9 折优惠（4860 元）。

**报名流程**：

1. 前置条件：孩子观察期评估结果为"通过"
2. 系统自动判断是否享受多孩优惠（同一用户已有 1 个正式会员孩子 → 价格自动变为 4860 元）
3. 点击"立即支付 5400 元"（或 4860 元）→ 调用微信支付
4. 支付成功 → 孩子状态变为正式会员（status=2）→ 会员有效期 365 天

**多孩优惠规则**：
- **资格判定条件**：同一 user 下已有 ≥1 个孩子状态为 观察期(1) 或 正式会员(2) → 该 user 的第 2 个孩子起享 9 折
- **覆盖范围**：适用于 观察期(500→450)、正式会员(5400→4860)、季度(1350→1215)、半年(2700→2430) 全部付费类型
- **优惠金额** = 原价 × `multi_child_discount`（默认 0.9），四舍五入到分
- **与续费折扣互斥**：多孩优惠与缓冲期续费折扣不可叠加；同时满足两项条件时，仅适用单重 9 折（不叠加为 8.1 折）
- **防刷规则**：已享受 9 折的订单不追溯补缴差价；该 user 的后续订单如不再满足资格条件（如所有老孩子退出），不再享多孩优惠
- 前端自动显示优惠价格，并提示"多孩优惠已应用"
- 价格表中"多孩折扣 0.9（9 折）"标注为可配置（配置键 `multi_child_discount`）

**前置条件校验**：
- 观察期评估未通过 → 提示"观察期评估未通过，暂不能报名正式会员"，无法进入报名页

### 1.3a 季度/半年会员

| 类型 | 价格 | 有效期 | 说明 |
|------|------|--------|------|
| 季度会员 | 1350 元 | 90 天 | 同正式会员权益 |
| 半年会员 | 2700 元 | 180 天 | 同正式会员权益 |
| 年费会员 | 5400 元 | 365 天 | 同正式会员权益 |

- 价格通过动态配置（price_quarterly / price_semi_annual）管理
- 三种会员类型均映射到 status=2（OFFICIAL），通过 Order.type 区分
- 到期后均自动转为 EXPIRED(3)
- 续费折扣同样适用（缓冲期内 9 折）

### 1.3b 会员升级

- 升级路径：季度 → 半年 → 年费
- 升级差价 = 目标价格 - 当前会员剩余价值
- 剩余价值 = 当前价格 ×（剩余天数 / 总天数）
- API：`GET /order/upgrade-options/{child_id}` 查询可升级选项
- API：`POST /order/upgrade` 创建升级订单

## 二、活动报名流程（activity_enrollment.feature）

### 2.1 活动浏览

**业务背景**：DmkWords 定期举办线下社群活动，增强用户粘性和品牌认同。活动类型包括：读书交流会、专家讲座、CityWalk 英文探索、郊游、年度大会等。

**浏览功能**：
- "即将举办"标签页：显示未来 30 天内的活动列表，每个活动显示封面、名称、时间、地点、报名状态（报名中/已满/已截止）
- "往期回顾"标签页：显示往期活动的照片、视频、总结文章

### 2.2 活动报名

**免费活动报名**：
1. 用户进入活动详情页，点击"立即报名"
2. 系统生成电子门票（ticket_code，格式：ACT-XXXXXXXX）
3. 报名记录存入 activity_enrollment 表（status=APPROVED）
4. 用户收到"活动报名成功"通知

**收费活动报名**：
1. 用户进入活动详情页，点击"立即报名"
2. 调用微信支付完成支付
3. 支付成功后生成电子门票
4. 订单状态为"已支付"

**名额校验**：
- 活动名额已满 → 按钮显示"名额已满"，无法点击报名
- 同一孩子重复报名同一活动 → 提示"已报名"

### 2.3 活动取消报名

**取消规则**：
- 活动开始前 24 小时以上：可取消
- 活动开始前 24 小时以内：不可取消，提示"活动开始前 24 小时内不可取消"

**免费活动取消**：
- 直接取消，报名状态变为"已取消"（STATUS_CANCELLED=3）

**收费活动取消**：
- 取消成功后自动申请全额退款
- 退款需管理员审核（status=PENDING）

**主办方取消活动**：
- 所有报名用户收到"活动取消"通知
- 付费用户自动全额退款，无需用户申请
- 退款在 3-7 个工作日内到账

### 2.4 活动签到

**签到流程**：
1. 活动开始后，用户出示电子门票二维码
2. 管理员扫描二维码 → 调用签到 API
3. 签到成功：报名状态变为"已签到"（STATUS_SIGNED_IN=4），记录签到时间
4. 电子门票显示"已签到"

### 2.5 活动提醒

- 活动开始前 3 天：系统自动发送"活动开始提醒"通知
- 通知推送到用户微信（订阅消息）
- 免费活动报名自动通过（APPROVED），收费活动待支付（PENDING→支付→APPROVED）
- 组织者取消活动 → 通知所有报名者 + 收费用户自动创建退款申请
- 活动开始前 3 天自动发送提醒通知（定时任务 check_activity_reminders）

## 三、会员权益转让（benefit_transfer.feature）

### 3.1 转让规则

**业务背景**：同一用户可以有多个孩子。如果一个孩子不再需要会员服务，可以将剩余权益转让给同一用户下的另一个孩子，避免资源浪费。

**转让条件**：

源孩子校验（5 项）：
1. 必须是观察期(1)或正式会员(2)
2. 必须已归还所有图书（无 BORROWING/OVERDUE 借阅记录）
3. 无未缴罚款（outstanding_fines == 0）
4. 无审核中的转让申请（无 PENDING 记录）
5. 该孩子此前未被转入过权益（无 APPROVED 记录 where target_child_id = 孩子ID）

目标孩子校验（3 项）：
1. 必须是体验用户（TRIAL=0）
2. 无活跃借阅（无 BORROWING/OVERDUE 记录）
3. 无未缴罚款（outstanding_fines == 0）

**转让流程**：
1. 家长提交转让申请
2. 管理员审核（通过/拒绝）
3. 审核通过 → 系统自动执行权益转移：
   - 目标孩子获得源孩子的剩余权益（会员状态 + 剩余天数）
   - 源孩子状态变为"已过期"
4. 审核拒绝 → 通知家长并附拒绝理由

**权益继承**：
- 正式会员（已使用 100 天）→ 目标孩子获得剩余 265 天正式会员
- 观察期（已使用 10 天）→ 目标孩子获得剩余 20 天观察期

> **T3.3 更新（2026-07-20）**：补全源孩子5项校验（新增"无审核中申请"和"未被转入过"）、目标孩子3项校验（D12确认目标仅限TRIAL=0）。

### 3.2 失败场景

- 源孩子有未归还图书 → 提示"有未归还图书，请先归还后再转让"
- 目标孩子已是正式会员 → 提示"目标孩子已是正式会员，无需转让"
- 源孩子状态不允许转让（如已退出）→ 拒绝操作

## 四、在线听读与阅读记录（online_reading.feature）

### 4.1 图书浏览

**业务背景**：线上图书馆是孩子选书的主要入口。图书支持按 AR 等级、主题、年龄等维度筛选。每本书显示封面、书名、作者、词数、预计阅读时间、是否有音频标识。

**筛选功能**：
- AR 等级筛选：如"AR2-AR3"，仅显示 AR 值在 2.0-3.9 之间的图书
- 主题筛选：如"故事"、"科学"、"历史"
- 年龄筛选：根据孩子年龄推荐适合的图书

**图书详情页**：
- 封面、简介、AR 等级、词数、预计阅读时间
- 是否有音频标识（音频齐备才显示"开始听读"按钮）
- 库存状态（显示"可借 X 本"或"暂无库存"）

### 4.2 音频伴读（核心功能）

**业务背景**：V3.5 彻底删除了线上 PDF 阅读器，改为音频伴读模式。孩子在线下借到实体书后，可以在线上播放该书的音频，配合实体书进行跟读学习。

**音频播放器功能**：
- 播放/暂停：点击按钮控制，音频从暂停位置继续播放
- 倍速播放：支持 0.75x / 1x / 1.25x / 1.5x
- 锁屏播放：使用 `wx.getBackgroundAudioManager()` 支持锁屏听音频
- 进度条：显示当前播放位置和总时长
- 中断恢复：退出小程序后重新打开，可以从上次音频位置继续听读

**技术要求**：
- 音频进度条更新严禁使用全局 `setData`，防止页面卡顿
- 翻页打点：监听 `onTimeUpdate`，结合后端 `audio_timeline` 实时高亮
- 逾期锁定：借阅超期后音频伴读功能锁死，无法播放

### 4.3 听读中查词

**查词流程**：
1. 孩子在听读过程中遇到不认识的单词
2. 点击文本中的单词 → 弹出查词弹窗
3. 显示：单词、音标、词性、中文释义、例句
4. 听读不中断（查词是浮层，不影响音频播放）

**查词数据来源**：
1. 优先从本地 ECDICT 词库查询（338 万词条，毫秒级响应）
2. 本地未命中 → 调用 Free Dictionary API 兜底
3. 查词结果自动记录到 user_vocabulary 表

### 4.4 阅读时长与打卡

**自动打卡规则**：
- 阅读满 10 分钟 → 自动触发阅读打卡（打卡类型="阅读时长"）
- 读完一本书 → 自动触发打卡（打卡类型="读完图书"）
- 每天最多打卡 1 次，不重复打卡

**阅读时长记录**：
- 每次听读会话记录：开始时间、结束时间、时长（秒）、阅读词数、阅读页数
- 累计阅读时间更新到 child.total_reading_minutes

## 五、书架/想读清单（bookshelf.feature）

### 5.1 书架定义（想读清单）

**V3.5 关键变更**：书架从"借阅书架"改为"想读清单/收藏夹"。书架与借阅完全分离，纯粹是"我想读这本书"的收藏行为，**不需要押金，没有数量限制**。

- 借阅功能由 `borrow_record` 表管理
- 书架只是线上收藏意图，容量无限

### 5.2 加入书架

**操作流程**：
1. 用户在图书详情页点击"加入书架"
2. 图书出现在书架列表中，书架数量 +1

**业务规则**：
- **不需要押金**：未缴纳押金的用户也可以加入书架
- **不限数量**：即使书架已有 50 本，仍可继续添加（V3.5 取消 20 本限制）
- **去重**：同一本书不能重复加入，提示"该书已在书架中"

### 5.3 移出书架

- 用户可以将图书从书架移除
- 移除后图书从列表消失，书架数量 -1

### 5.4 书架浏览

- 显示所有已加入的图书：封面、标题
- 每本书显示阅读进度（如有阅读记录）

### 5.5 从书架预约借书

**业务流程**：
1. 用户在书架中选择一本想借的书
2. 点击"预约借书"
3. 前置条件：已缴纳押金 + 该书库存 > 0
4. 创建预约记录 → 库存锁定 → 显示"预约成功，请 72 小时内到店取书"

## 六、实体书借阅（borrow_record.feature）

### 6.1 扫码借书

**业务背景**：线下门店工作人员使用手持扫码枪扫描实体书条码，系统自动创建借阅记录。这是 OMO 模式的核心线下环节。

**借书流程**：

**已有馆藏的书**：
1. 工作人员扫描条码（如 978-0-06-112495-1-001）
2. 系统识别到该条码对应的 BookCopy → 关联到 Book
3. 创建借阅记录（BorrowRecord）：status=BORROWING, due_date=当前+21天
4. 发布 book.borrowed 事件 → BookCopy 状态变为 BORROWED

**首次扫码（条码不存在）**：
1. 工作人员扫描条码 → 系统提示"条码不存在，请输入图书信息"
2. 工作人员输入书名和 ISBN → 确认创建
3. 系统自动：创建 Book 记录 → 创建 BookCopy 记录（关联条码）→ 创建借阅记录

**同 ISBN 不同条码**：
1. 工作人员扫描新条码 → 系统识别到 ISBN 已存在
2. 仅创建新的 BookCopy（关联到已有 Book）→ 创建借阅记录
3. 不重复创建图书记录

**借书权限校验**：
- 孩子必须是观察期或正式会员（status=1 或 2）
- 必须已缴纳押金（deposit_status=PAID）
- 同时借阅数不超过上限（默认 20 本）
- 同一本书不能重复借（同一 child+book 已有 BORROWING 记录）

### 6.2 还书

**正常还书**：
1. 工作人员扫描条码 → 匹配到借阅记录
2. 借阅记录状态更新为 RETURNED，归还时间为当前时间
3. 发布 book.returned 事件 → BookCopy 状态恢复 AVAILABLE → 库存释放

**逾期还书**：
1. 还书时系统检测到当前时间 > due_date
2. 计算逾期天数 = (当前时间 - due_date).days
3. 生成逾期罚款记录：罚款 = 逾期天数 × 1 元/天（可配置）
4. 借阅记录状态仍为 RETURNED，但记录 fine_amount

**丢失图书**：
1. 工作人员标记该书为"丢失"
2. 罚款金额 = 图书定价 × 1.5 倍
3. 从押金中扣除罚款金额
4. 更新孩子的 outstanding_fines

- 扫码借书端点：POST /borrow/scan（条码存在直接借阅，不存在自动创建图书+副本+借阅）
- 扫码还书端点：POST /borrow/scan-return（通过条码找到活跃借阅记录并还书）
- 丢书登记端点：POST /admin/borrow-records/{record_id}/mark-lost
- 状态转换：BORROWING/OVERDUE → LOST(3)
- 罚款金额 = 图书定价 × lost_book_fine_multiplier（默认 1.5，可配置）
- 库存扣减：total_stock - 1（实体书丢失不可恢复）

### 6.3 逾期检测（定时任务）

**每日凌晨 2 点执行**：
1. 查询所有 status=BORROWING 且 due_date < 当前时间的记录
2. 标记状态为 OVERDUE
3. 发布 book.overdue 事件 → 孩子的在线音频伴读功能锁定

### 6.4 到期提醒（定时任务）

**提醒节点**（可配置，默认 5/3/1/当天）：
- 到期前 5 天：发送微信订阅消息提醒
- 到期前 3 天：发送提醒
- 到期前 1 天：发送提醒
- 到期当天：发送提醒，提示"今天到期，请尽快归还"

## 七、押金管理（deposit.feature）

### 7.1 缴纳押金

**业务背景**：押金是线下借阅的前置条件。家长为孩子缴纳 1200 元押金后，孩子才能借阅实体书。押金在孩子退出会员时可退还（需满足条件）。

**缴纳流程**：
1. 家长在押金页面点击"缴纳押金 1200 元"
2. 调用微信支付完成支付
3. 创建 DepositRecord（status=PAID, amount=1200）
4. 更新 child.deposit_status = PAID

**防重复缴纳**：
- 已缴纳押金的孩子再次缴纳 → 提示"押金已缴纳，无需重复操作"

### 7.2 押金退款

**退款条件**（必须同时满足）：
1. 无活跃借阅记录（所有 BorrowRecord 状态不为 BORROWING）
2. 无未缴罚款（child.outstanding_fines == 0）

**退款流程**：
1. 家长申请退款
2. 系统校验退款条件
3. 条件满足 → 创建退款申请 → 押金状态更新为"退款中"
4. 管理员审核 → 审核通过 → 退款在 3-7 个工作日内原路退回

**退款拒绝场景**：
- 有活跃借阅 → 提示"请先归还所有借阅图书"
- 有未结罚款 → 提示"请先结清未缴罚款 XX 元"

### 7.3 押金扣除

**丢书扣除**：
- 罚款金额 = 图书定价 × 1.5 倍
- 从押金余额中扣除
- 更新 child.outstanding_fines

**逾期罚款扣除**：
- 逾期罚款 = 逾期天数 × 每天罚款金额（默认 1 元/天）
- 从押金余额中扣除
- 更新 child.outstanding_fines

### 7.4 押金查询

家长可以查看：
- 押金状态：未缴纳(0) → 已缴纳(1) → 退款中(4) → 已退(2) / 已扣(3)
- 押金余额
- 未结罚款金额

### 7.5 多孩押金规则（D6/D7）

**多孩押金**（D6=A）：
- 多孩家庭押金**无优惠**，每个孩子独立缴纳 1200 元（与会员费多孩 9 折为独立策略；押金是履约担保而非消费，不参与折扣）
- 已缴纳押金的孩子不影响其他未缴纳孩子的状态

**欠费连坐**（D7=A）：
- **孩子各自独立**：outstanding_fines 按孩子独立记录，不跨孩子连坐
- 孩子 A 有未缴罚款 → 不影响孩子 B 的借阅、退款、报名活动
- 退款拦截网（15.1）只校验**本孩子的**活跃借阅与未缴罚款
- 押金退款条件（7.2）只校验**本孩子的**无活跃借阅 + 无未缴罚款

**商业风险提示**：
- 3 孩家庭预付超 13000 元（押金 3600 + 会员费），需运营验证该门槛对多孩转化率的影响
- 若运营反馈负面，D6 商业策略（如押金分期、家庭押金池）需重开决策

## 十一、晋级体系（advancement.feature）

### 11.1 级别管理

**级别定义**：
- A-Z 共 26 级，每级有名称、徽章 Emoji、排序序号
- 每级默认需读完 5 本书（可按级别单独配置）
- 测验最低通过率默认 80%（可配置）
- 每级有最大可读 AR 等级限制

**初始级别分配**：
- 孩子首次注册时自动分配最低级别
- 显示当前级别徽章

**级别信息展示**：
- 当前级别名称和徽章
- 晋级要求：需读完几本书、测验通过率
- 当前进度：已读完几本、已通过几次测验

### 11.2 阅读提交

**提交流程**：
1. 孩子完成听读（读完最后一页）→ 自动创建阅读提交记录（ReadingSubmission）
2. 提交状态为"待审核"（STATUS_PENDING）
3. 老师收到待审核通知

**老师审核**：
- 审核通过 → 提交状态变为"审核通过"（STATUS_APPROVED）→ 该级别已读完书数 +1
- 审核打回 → 提交状态变为"审核拒绝"（STATUS_REJECTED）→ 孩子收到打回通知

### 11.3 测验

**题库管理**：
- 每本书至少 5 道选择题（A/B/C/D 四选一）
- 每题有正确答案、解析、难度（1-5）
- 老师负责创建和维护题库

**出卷流程**：
1. 老师为孩子出测验 → 从题库中抽取题目
2. 创建测验实例（Quiz）：status=IN_PROGRESS, total_questions=题目数

**答题流程**：
1. 孩子逐题作答
2. 提交答案 → 系统评分
3. 测验状态变为 COMPLETED，记录 correct_count 和 score

### 11.4 测验结果

**通过条件**：
- 正确率 ≥ 80%（可配置）→ 测验通过
- 通过后：借阅记录的测评状态更新为 PASSED

**未通过**：
- 正确率 < 80% → 测验未通过
- 借阅记录的测评状态保持 PENDING
- 显示"可以重考"提示

**积分计入**：
- 测验通过后，该书的 word_count 计入孩子积分（child.total_words_read）
- **积分去重**：同一 child+book 只计入一次 word_count，重复测评不重复计分

### 11.5 晋级

**晋级条件**（同时满足）：
1. 本级已读完足够书数（books_read_at_level >= required_books）
2. 本级已通过足够次测验（quizzes_passed_at_level >= quiz_pass_count）

**两个阈值均可通过系统配置动态调整**：
- `required_books`：每级默认 5 本（可按级别单独配置）
- `quiz_pass_count`：每级需通过测验的最少书数，默认 5（全部通过），可配为 1-5

**晋级流程**：
1. 系统检测晋级条件 → 条件满足
2. 当前级别标记为非当前（is_current=False）
3. 创建新级别记录（is_current=True）
4. 发布 level.advanced 事件 → 自动生成晋级证书
5. 获得新级别徽章
6. 借阅上限增加（max_borrow_count）

**最高级别**：
- 已是最高级别时尝试晋级 → 显示"已是最高级别"提示

### 11.6 成就系统

**成就类型**：
- 晋级成就：首次晋级、达到特定级别
- 读书里程碑：读完 10/50/100 本书
- 连续打卡：连续 7/21/100 天打卡
- 测验满分：获得 100% 正确率

**成就展示**：
- 成就页面显示所有已获得的徽章（彩色）
- 未获得的徽章显示为灰色
- 排行榜：按阅读量降序排列，前 3 名金银铜牌

## 十五、退款申请流程（refund_application.feature）

### 15.1 退款规则

**亲子课退款**：
- 课程开始前：全额退款 99 元
- 课程开始后：不可退款

**观察期退款**：
- 退款金额 = 实付金额 - (实付金额 ÷ 30 × 已使用天数)
- 示例：实付 500 元，已使用 10 天 → 退款 = 500 - 500÷30×10 = 333.33 元
- 多孩优惠：按实付金额计算（如 9 折实付 450 元 → 450 - 450÷30×15 = 225 元）

**正式会员退款**：
- 退款金额 = 实付金额 - (实付金额 ÷ 365 × 已使用天数)
- 示例：实付 5400 元，已使用 30 天 → 退款 = 5400 - 5400÷365×30 = 4956.16 元
- 多孩优惠：按实付金额计算

**通用规则**：
- 退款金额由服务端计算，不信任客户端
- 退款在 3-7 个工作日内原路退回
- 退款拦截网：申请退款前校验孩子名下无活跃借阅记录（BORROWING/OVERDUE），否则拒绝退款

### 15.2 退款审核

**审核流程**：
1. 用户提交退款申请
2. 管理员审核（通过/拒绝）
3. 审核通过 → 系统自动发起退款 → 用户收到"退款审核通过"通知
4. 审核拒绝 → 用户收到"退款审核拒绝"通知，包含拒绝理由

## 十七、RBAC 管理后台权限控制（Phase 1-5）

### 17.1 角色体系

管理后台有 3 级 RBAC 角色，由 `role` 表定义：

| 角色 | 权限数 | 典型操作 |
|------|--------|---------|
| super_admin（超级管理员） | 128 | 全部权限，含退款审批/押金扣除/全局配置/管理员 CRUD/角色管理 |
| staff（运营人员） | 102 | 扫码借还/录入图书/发起退款/审核/报告/数据统计/消息/活动/场馆 |
| teacher（教师） | 27 | 查看用户/孩子/图书/订单/报告 + 审核提交 + 活动签到 |

### 17.2 权限分配

- 权限码格式：`group.action`（如 `user.create`、`order.refund`）
- 每个权限码对应一个 API 端点或页面
- 种子共 128 个权限，覆盖 31 个功能分组
- 角色-权限关联通过 `role_permission` 表实现（多对多）

### 17.3 管理界面

- **角色管理**（`/admin/view/roles`）：角色列表 → 点击"编辑权限" → 按分组展示权限 checkbox 树 → 全量替换保存
- **管理员账号**（`/admin/view/settings` 管理员 Tab）：角色从 API 动态加载，可选运营人员/教师/超级管理员
- **老师管理**（`/admin/view/teachers`）：老师卡片底部显示管理员状态，可直接创建/编辑关联的管理员账号

### 17.4 权限检查体系

- **后端 API**：`require_perm("xxx.xxx")` 依赖注入，每个端点单独指定
- **页面访问**：`PAGE_PERM_MAP` 统一守卫，无权限重定向 403
- **前端按钮**：`data-perm` 属性 + `applyPermissions()` 运行时裁剪
- **数据隔离**：teacher 角色通过 `get_scoped_child_ids()` 仅返回自己负责的孩子数据

### 17.5 变更审计

- 角色权限变更通过 `OperationLog` 记录（module="role"，operation="update_permissions"）
- 管理员创建/更新通过 `OperationLog` 记录（前端采集 + 后端写入）

## 附录D：库存双口径联动矩阵

### D.1 BookCopy 六类状态枚举

| 编码 | 常量名 | 含义 | 备注 |
|------|--------|------|------|
| 0 | AVAILABLE | 可借（在馆） | 初始状态 |
| 1 | BORROWED | 已借出 | 借书 / 预约取书时转入 |
| 2 | MAINTENANCE | 维修中 | 管理员手动标记（目前无操作入口） |
| 3 | SCRAPPED | 报废 | 管理员手动标记（目前无操作入口） |
| 4 | DAMAGED | 损坏 | T3.6a 引入，三级定级后标记 |
| 5 | LOST | 丢失 | T3.6a 引入，赔偿后标记 |

### D.2 BookCopy 状态变更 → Book 库存联动矩阵

每本 Book 的 `total_stock`/`available_stock` 为**汇总口径**，由该 Book 下所有 BookCopy 的状态派生而来。以下矩阵定义每类状态变更对 Book 库存字段的原子影响：

| # | 操作 | BookCopy 旧状态 | BookCopy 新状态 | total_stock | available_stock |
|---|------|----------------|-----------------|-------------|-----------------|
| 1 | 新增副本 | — | AVAILABLE(0) | +1 | +1 |
| 2 | 借出 | AVAILABLE(0) | BORROWED(1) | 不变 | -1 |
| 3 | 还书 | BORROWED(1) | AVAILABLE(0) | 不变 | +1 |
| 4 | 预约创建 | —（预约不占具体副本） | — | 不变 | -1 |
| 5 | 预约取消/过期 | — | — | 不变 | +1 |
| 6 | 标记维修(在馆) | AVAILABLE(0) | MAINTENANCE(2) | 不变 | -1 |
| 7 | 标记维修(已借出) | BORROWED(1) | MAINTENANCE(2) | 不变 | 不变 |
| 8 | 维修完成 | MAINTENANCE(2) | AVAILABLE(0) | 不变 | +1 |
| 9 | 标记损坏(在馆) | AVAILABLE(0) | DAMAGED(4) | 不变 | -1 |
| 10 | 标记损坏(已借出) | BORROWED(1) | DAMAGED(4) | 不变 | 不变 |
| 11 | 损坏修复 | DAMAGED(4) | AVAILABLE(0) | 不变 | +1 |
| 12 | 丢失赔偿 | BORROWED(1) | LOST(5) | -1 | -1（已扣，无实际影响） |
| 13 | 报废(在馆) | AVAILABLE(0) | SCRAPPED(3) | -1 | -1 |
| 14 | 报废(已借出) | BORROWED(1) | SCRAPPED(3) | -1 | 不变 |
| 15 | 报废(维修中) | MAINTENANCE(2) | SCRAPPED(3) | -1 | 不变 |
| 16 | 报废(损坏) | DAMAGED(4) | SCRAPPED(3) | -1 | 不变 |

**口径规则**：
- `total_stock` = 该 Book 下状态为 AVAILABLE + BORROWED + MAINTENANCE + DAMAGED 的 BookCopy 数量（SCRAPPED 和 LOST 不计入）
- `available_stock` = 该 Book 下状态为 AVAILABLE 的 BookCopy 数量
- 所有库存变更必须使用 SQL 原子操作（`UPDATE ... SET field = field +/- N`），避免并发超卖

### D.3 代码审计偏差报告（2026-07-20）

| # | 偏差 | 严重程度 | 说明 |
|---|------|---------|------|
| D01 | BookCopyStatus 缺 DAMAGED/LOST 两状态 | 高 | 已在 backend/common/types.py 补全为六类枚举。实际使用者为 T3.6a。 |
| D02 | 维修/报废无管理员操作入口 | 中 | MAINTENANCE(2) / SCRAPPED(3) 状态在 BookCopy 模型中已定义 4 年，但后台没有对应菜单和 API。不是本轮修复范围。 |
| D03 | 预约创建只校验 available_stock > 0，不锁定具体副本 | 低 | 预约不锁定具体 BookCopy（取书时才分配），在预约量 < 库存量时不会超卖。但在高并发下预约数量可能短时超过库存。当前设计可接受。 |
| D04 | `book/service.py add_copies()` 和 `borrow/service.py create_book()` 各有一套库存递增逻辑，重复 | 低 | 两个入口服务于不同场景：admin 后台 vs 扫码录入。逻辑等价，暂不合。 |
| D05 | 丢书赔偿后 (`deposit/service.py`) 未更新 BookCopy.status → LOST | 高 | 丢书扣款后 BookCopy.status 仍为 BORROWED(1)，未标记 LOST(5)。需 T3.6a 落地后联动。 |

### D.4 每日对账任务定义

**任务名称**：`daily_stock_reconciliation`

**触发时间**：每天凌晨 3:00

**逻辑**：

```python
def reconcile_stock():
    """每日库存对账：Book.total_stock/available_stock 与 BookCopy 实际计数对比"""
    for book in all_books:
        copy_count = SELECT COUNT(*) FROM book_copy
            WHERE book_id = :book_id AND is_deleted = 0
        total_count = SELECT COUNT(*) FROM book_copy
            WHERE book_id = :book_id AND is_deleted = 0
            AND status IN (AVAILABLE, BORROWED, MAINTENANCE, DAMAGED)
        avail_count = SELECT COUNT(*) FROM book_copy
            WHERE book_id = :book_id AND is_deleted = 0
            AND status = AVAILABLE
        
        if book.total_stock != total_count OR book.available_stock != avail_count:
            # 记录偏差到 operation_log
            INSERT INTO operation_log(
                action='stock_reconciliation',
                target_type='book',
                target_id=book.id,
                detail=json(...),
            )
            # 自动修正（静默对齐）
            book.total_stock = total_count
            book.available_stock = avail_count
```

**部署**：在 `backend/tasks/scheduler.py` 的 `init_scheduler()` 中注册。

### 角色生命周期管理

- **创建**：支持创建自定义角色，可从 staff 或 teacher 模板快照复制权限（非引用，独立副本）。
  - 角色名必填（≤20 字），角色代码必填（小写字母/数字/下划线，唯一，创建后不可修改）。
  - 创建成功后自动打开该角色的权限配置弹窗。
- **重命名**：支持修改角色名称和描述。系统内置角色（is_system=1）禁止改名。
- **删除**：支持删除自定义角色。系统内置角色不可删除；有管理员账号引用的角色不可删除（返回错误提示）。
- **最后一个启用超管保护**：禁用、删除或降权操作时，自动校验操作后启用状态的超级管理员数量 ≥ 1，否则拒绝操作（返回"至少需要保留一个启用的超级管理员"）。

## 附录I：多角色操作定义

### I.1 角色与登录态
- 家长：微信小程序唯一登录主体，持有微信 openid 会话。所有支付、退款、押金、权益转让操作必须由家长完成。
- 孩子：无独立登录态，是家长账号下的数据实体。家长通过"当前选中孩子"（user.current_child_id）切换操作对象。
- 孩子侧所有操作（听读、查词、朗读、答题）均以"家长设备 + 当前选中孩子"执行，产生的数据归属当前选中孩子。

### I.2 多孩子切换规则
- 切换入口：小程序首页顶部孩子头像，点击展开孩子列表。
- 切换后：所有页面数据（书架、统计、生词本、晋级）立即切换为选中孩子的数据，无缓存残留。
- 切换时机限制：支付流程中禁止切换孩子（防止订单归属错乱）；听读进行中切换孩子，当前听读会话强制结束并结算时长（前端已实现/待验证）。

### I.3 年龄差异化
- 本版本不做年龄差异化 UI。所有孩子共用同一套交互。3-6 岁孩子由家长辅助操作，属使用场景假设，不构成系统功能需求。
