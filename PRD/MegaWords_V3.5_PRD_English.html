1. User Enrollment Flow (user_enrollment.feature)

1.1 Parent-Child Course Enrollment (99 yuan)

Business Background: The parent-child course is MegaWords' customer acquisition entry point. Parents pay 99 yuan (CNY) and bring their child to a physical store for a 75-minute English reading experience session. After the session, the teacher issues an English reading proficiency assessment report for the child, which serves as a prerequisite for subsequent observation period enrollment.

Enrollment Flow:

  1. Parent selects a venue on the mini-program homepage (e.g., People's Square Branch / Xuhui Branch)
  2. Selects a class time (fixed time slots on Saturday/Sunday, limited to 30 participants per session)
  3. Fills in information: parent's name, phone number (auto-filled via WeChat authorization), child's Chinese name, English name, age (3-15 years), grade level
  4. Clicks "Pay 99 yuan Now" → calls WeChat Pay V3
  5. Payment successful → order generated (type=1, pay_status=1) → electronic voucher generated

Validation Rules:

  • All fields are required; otherwise the payment button is grayed out
  • Selected time slot is full → prompt: "This time slot is fully booked, please select another time"
  • Phone number is auto-filled via WeChat authorization, manual entry is not required

Data Flow: Payment successful → create Order (type=PARENT_COURSE) → publish order.paid event

1.2 Observation Period Enrollment (500 yuan (CNY) / 30 days)

Business Background: After the parent-child course, if parents are satisfied with the teaching quality, they can enroll their child in the observation period. The observation period is a complete 30-day trial experience where the child enjoys exactly the same services as a full member (with no feature restrictions), including: unlimited book reading, audio read-along, word lookup, assessments, check-ins, and leaderboards. At the end of the observation period, the system automatically generates a detailed reading report to guide parents toward converting to full membership.

Enrollment Flow:

  1. Prerequisite: The child has completed the parent-child course and received an assessment report
  2. Parent selects the observation period start date (defaults to the current day)
  3. Clicks "Pay 500 yuan Now" → calls WeChat Pay
  4. Payment successful → order generated (type=OBSERVATION) → publish order.paid event
  5. Event handler automatically: child status changes to observation period (status=1), one-on-one mentor assigned (teacher_id)

Key Rules:

  • Observation period users enjoy exactly the same services as full members, with no feature restrictions
  • Observation period is a fixed 30 days, starting from the payment date
  • System automatically sends renewal reminders at 7/5/3/2/1 day(s) before expiration and on the expiration day
  • An observation period report is automatically generated upon expiration (number of books read, word count, number of assessments passed, teacher comments)
  • After expiration, a 15-day grace period begins; renewals during this period receive a 10% discount

Prerequisite Validation:

  • Child has not completed the parent-child course → prompt: "Please complete the parent-child course and obtain an assessment report first," unable to access the enrollment page

1.3 Full Membership Enrollment (5,400 yuan (CNY) / year)

Business Background: After the observation period ends, parents can choose to enroll their child as a full member. Full membership is valid for 365 days and includes all services. Starting from the second child of the same user, a 10% discount (4,860 yuan (CNY)) is available.

Enrollment Flow:

  1. Prerequisite: The child's observation period evaluation result is "Passed"
  2. System automatically determines eligibility for the multi-child discount (if the same user already has 1 full member child → price automatically changes to 4,860 yuan (CNY))
  3. Clicks "Pay 5,400 yuan Now" (or 4,860 yuan (CNY)) → calls WeChat Pay
  4. Payment successful → child status changes to full member (status=2) → membership valid for 365 days

Multi-Child Discount Rules:

  • If the same user already has 1 child with observation period or full membership status → the second child onward receives a 10% discount
  • Discount amount = original price × 0.9, rounded to the nearest cent
  • Frontend automatically displays the discounted price with the prompt: "Multi-child discount applied"

Prerequisite Validation:

  • Observation period evaluation not passed → prompt: "Observation period evaluation not passed, unable to enroll as full member at this time," unable to access the enrollment page

1.3a Quarterly / Semi-Annual Membership

Type Price Validity Description
Quarterly Membership 1,350 yuan (CNY) 90 days Same privileges as full membership
Semi-Annual Membership 2,700 yuan (CNY) 180 days Same privileges as full membership
Annual Membership 5,400 yuan (CNY) 365 days Same privileges as full membership
  • Prices are managed through dynamic configuration (price_quarterly / price_semi_annual)
  • All three membership types map to status=2 (OFFICIAL), distinguished by Order.type
  • All automatically transition to EXPIRED(3) upon expiration
  • Renewal discounts also apply (10% discount during the grace period)

1.3b Membership Upgrade

  • Upgrade path: Quarterly → Semi-Annual → Annual
  • Upgrade price difference = target price − current membership remaining value
  • Remaining value = current price × (remaining days / total days)
  • API: GET /order/upgrade-options/{child_id} to query available upgrade options
  • API: POST /order/upgrade to create an upgrade order

2. Activity Enrollment Flow (activity_enrollment.feature)

2.1 Activity Browsing

Business Background: MegaWords regularly hosts offline community activities to enhance user engagement and brand loyalty. Activity types include: book exchange meetups, expert lectures, CityWalk English exploration, outings, annual conferences, and more.

Browsing Features:

  • "Upcoming" tab: Displays a list of activities within the next 30 days. Each activity shows its cover image, name, time, location, and registration status (Open for Registration / Fully Booked / Registration Closed)
  • "Past Events" tab: Displays photos, videos, and summary articles from past activities

2.2 Activity Registration

Free Activity Registration:

  1. User enters the activity detail page and clicks "Register Now"
  2. System generates an electronic ticket (ticket_code, format: ACT-XXXXXXXX)
  3. Registration record is stored in the activity_enrollment table (status=APPROVED)
  4. User receives an "Activity Registration Successful" notification

Paid Activity Registration:

  1. User enters the activity detail page and clicks "Register Now"
  2. Calls WeChat Pay to complete the payment
  3. Electronic ticket is generated after successful payment
  4. Order status is set to "Paid"

Capacity Validation:

  • Activity is fully booked → button displays "Fully Booked" and cannot be clicked for registration
  • Same child registers for the same activity again → prompt: "Already Registered"

2.3 Activity Registration Cancellation

Cancellation Rules:

  • More than 24 hours before the activity starts: Cancellation is allowed
  • Less than 24 hours before the activity starts: Cancellation is not allowed; prompt: "Cancellation is not permitted within 24 hours before the activity starts"

Free Activity Cancellation:

  • Cancelled directly; registration status changes to "Cancelled" (STATUS_CANCELLED=3)

Paid Activity Cancellation:

  • After successful cancellation, a full refund is automatically requested
  • Refund requires administrator approval (status=PENDING)

Organizer Cancels Activity:

  • All registered users receive an "Activity Cancelled" notification
  • Paid users receive an automatic full refund without needing to submit a request
  • Refunds are credited within 3-7 business days

2.4 Activity Check-In

Check-In Flow:

  1. After the activity begins, the user presents the electronic ticket QR code
  2. Administrator scans the QR code → calls the check-in API
  3. Check-in successful: registration status changes to "Checked In" (STATUS_SIGNED_IN=4), check-in time recorded
  4. Electronic ticket displays "Checked In"

2.5 Activity Reminders

  • 3 days before the activity starts: system automatically sends an "Activity Starting Soon" reminder notification
  • Notifications are pushed to the user's WeChat (subscription messages)
  • Free activity registrations are automatically approved (APPROVED); paid activities are pending payment (PENDING → payment → APPROVED)
  • Organizer cancels activity → notify all registrants + automatically create refund requests for paid users
  • Automatic reminder notifications sent 3 days before the activity starts (scheduled task: check_activity_reminders)

3. Membership Benefit Transfer (benefit_transfer.feature)

3.1 Transfer Rules

Business Background: A single user can have multiple children. If one child no longer needs membership services, the remaining benefits can be transferred to another child under the same user account to avoid resource waste.

Transfer Conditions:

  • The source child must be in the observation period or full membership
  • The source child must have returned all books (no active borrowing records)
  • The target child must not already be a full member (otherwise the transfer would be meaningless)
  • Transfers are only allowed between children under the same user account

Transfer Flow:

  1. Parent submits a transfer request
  2. Administrator reviews the request (Approve / Reject)
  3. Approval → system automatically executes the benefit transfer:
    • Target child receives the source child's remaining benefits (membership status + remaining days)
    • Source child's status changes to "Expired"
  4. Rejection → parent is notified with the rejection reason

Benefit Inheritance Examples:

  • Full membership (100 days used) → target child receives the remaining 265 days of full membership
  • Observation period (10 days used) → target child receives the remaining 20 days of observation period

3.2 Failure Scenarios

  • Source child has unreturned books → prompt: "There are unreturned books. Please return them before transferring."
  • Target child is already a full member → prompt: "The target child is already a full member. Transfer is unnecessary."
  • Source child's status does not allow transfer (e.g., has withdrawn) → operation is rejected

5. Bookshelf / Wishlist (bookshelf.feature)

5.1 Bookshelf Definition (Wishlist)

V3.5 Key Change: The bookshelf has been changed from a "borrowing shelf" to a "wishlist / favorites". The bookshelf is completely separated from borrowing — it is purely a "I want to read this book" collection action, requires no deposit, and has no quantity limit.

  • Borrowing is managed by the borrow_record table
  • The bookshelf is an online collection intent with unlimited capacity

5.2 Adding to Bookshelf

Operation Flow:

  1. User taps "Add to Bookshelf" on the book detail page
  2. The book appears in the bookshelf list; the bookshelf count increases by 1

Business Rules:

  • No deposit required: Users who have not paid a deposit can still add books to the bookshelf
  • No quantity limit: Even if the bookshelf already has 50 books, more can be added (V3.5 removes the 20-book limit)
  • Deduplication: The same book cannot be added twice; a prompt "This book is already on your bookshelf" is shown

5.3 Removing from Bookshelf

  • Users can remove books from the bookshelf
  • After removal, the book disappears from the list; the bookshelf count decreases by 1

5.4 Bookshelf Browsing

  • Displays all added books: cover, title
  • Each book shows reading progress (if a reading record exists)

5.5 Reserving a Book from the Bookshelf

Business Flow:

  1. User selects a book they want to borrow from the bookshelf
  2. Taps "Reserve Book"
  3. Prerequisites: deposit paid + book inventory > 0
  4. A reservation record is created → inventory is locked → displays "Reservation successful. Please pick up the book in-store within 72 hours"

6. Physical Book Borrowing (borrow_record.feature)

6.1 Scan to Borrow

Business Context: Offline store staff use handheld barcode scanners to scan physical book barcodes, and the system automatically creates borrowing records. This is the core offline component of the OMO model.

Borrowing Flow:

Books Already in the Collection:

  1. Staff scans the barcode (e.g., 978-0-06-112495-1-001)
  2. System identifies the BookCopy associated with the barcode → links it to a Book
  3. A BorrowRecord is created: status=BORROWING, due_date=current date + 21 days
  4. A book.borrowed event is published → BookCopy status changes to BORROWED

First Scan (Barcode Not Found):

  1. Staff scans the barcode → system prompts "Barcode not found. Please enter book information"
  2. Staff enters the book title and ISBN → confirms creation
  3. System automatically: creates a Book record → creates a BookCopy record (linked to the barcode) → creates a borrowing record

Same ISBN, Different Barcode:

  1. Staff scans a new barcode → system recognizes the ISBN already exists
  2. Only creates a new BookCopy (linked to the existing Book) → creates a borrowing record
  3. A duplicate book record is not created

Borrowing Permission Checks:

  • The child must be in the trial period or a formal member (status=1 or 2)
  • A deposit must have been paid (deposit_status=PAID)
  • Simultaneous borrows must not exceed the limit (default 20 books)
  • The same book cannot be borrowed twice (an existing BORROWING record for the same child+book)

6.2 Returning Books

Normal Return:

  1. Staff scans the barcode → matches it to the borrowing record
  2. Borrowing record status is updated to RETURNED; return time is set to the current time
  3. A book.returned event is published → BookCopy status is restored to AVAILABLE → inventory is released

Overdue Return:

  1. At the time of return, the system detects that the current time > due_date
  2. Overdue days are calculated = (current time - due_date).days
  3. An overdue fine record is generated: fine = overdue days × 1 yuan/day (CNY) (configurable)
  4. Borrowing record status remains RETURNED, but the fine_amount is recorded

Lost Book:

  1. Staff marks the book as "Lost"
  2. Fine amount = book price × 1.5 times
  3. The fine is deducted from the deposit
  4. The child's outstanding_fines is updated
  • Scan-to-borrow endpoint: POST /borrow/scan (if barcode exists, borrows directly; if not, automatically creates Book + BookCopy + BorrowRecord)
  • Scan-to-return endpoint: POST /borrow/scan-return (finds the active borrowing record via barcode and returns the book)
  • Lost book registration endpoint: POST /admin/borrow-records/{record_id}/mark-lost
  • State transitions: BORROWING/OVERDUE → LOST(3)
  • Fine amount = book price × lost_book_fine_multiplier (default 1.5, configurable)
  • Inventory deduction: total_stock - 1 (lost physical books are not recoverable)

6.3 Overdue Detection (Scheduled Task)

Executes daily at 2:00 AM:

  1. Queries all records where status=BORROWING and due_date < current time
  2. Marks status as OVERDUE
  3. Publishes a book.overdue event → the child's online audio-guided reading function is locked

6.4 Due Date Reminders (Scheduled Task)

Reminder Milestones (configurable, default 5/3/1/same day):

  • 5 days before due date: sends a WeChat subscription message reminder
  • 3 days before due date: sends a reminder
  • 1 day before due date: sends a reminder
  • On the due date: sends a reminder with "Due today. Please return as soon as possible"

7. Deposit Management (deposit.feature)

7.1 Paying the Deposit

Business Context: The deposit is a prerequisite for offline borrowing. Parents pay a 1,200 yuan (CNY) deposit for their child, after which the child can borrow physical books. The deposit is refundable when the child leaves the membership (subject to conditions being met).

Payment Flow:

  1. Parent taps "Pay 1,200 yuan (CNY) deposit" on the deposit page
  2. Completes payment via WeChat Pay
  3. A DepositRecord is created (status=PAID, amount=1200)
  4. child.deposit_status is updated to PAID

Duplicate Payment Prevention:

  • If a parent tries to pay again for a child who already has a paid deposit → prompt "Deposit already paid. No need to repeat."

7.2 Deposit Refund

Refund Conditions (all must be satisfied simultaneously):

  1. No active borrowing records (no BorrowRecord with status BORROWING)
  2. No unpaid fines (child.outstanding_fines == 0)

Refund Flow:

  1. Parent requests a refund
  2. System validates the refund conditions
  3. Conditions met → a refund request is created → deposit status is updated to "Refund in Progress"
  4. Admin review → approval granted → refund is processed back to the original payment method within 3-7 business days

Refund Rejection Scenarios:

  • Active borrowings exist → prompt "Please return all borrowed books first"
  • Unpaid fines exist → prompt "Please settle the unpaid fine of XX yuan (CNY) first"

7.3 Deposit Deductions

Lost Book Deduction:

  • Fine amount = book price × 1.5 times
  • Deducted from the deposit balance
  • child.outstanding_fines is updated

Overdue Fine Deduction:

  • Overdue fine = number of overdue days × daily fine amount (default 1 yuan/day (CNY))
  • Deducted from the deposit balance
  • child.outstanding_fines is updated

7.4 Deposit Inquiry

Parents can view:

  • Deposit status: Unpaid(0) → Paid(1) → Refund in Progress(4) → Refunded(2) / Deducted(3)
  • Deposit balance
  • Outstanding fine amount

8. Online Book Reservation (reservation.feature)

8.1 Reservation Flow

Business Background: Parents can reserve physical books online and pick them up directly at the store, saving wait time.

Reservation Flow:

  1. User clicks "Reserve Book" on the book detail page
  2. Prerequisites: deposit paid + stock > 0
  3. Create a Reservation (status=PENDING)
  4. Stock is locked (available_stock -1)
  5. Reservation expiration time = current time + 72 hours
  6. Display "Reservation successful, please pick up the book in-store within 72 hours"

Reservation Validation:

  • Deposit not paid → prompt "Please pay the deposit first"
  • Stock is 0 → prompt "This book is currently out of stock"

8.2 Reservation Pickup

Pickup Flow:

  1. User arrives at the store, staff scans the barcode
  2. System matches the user's reservation record
  3. A formal borrowing record is created (due_date=current date + 21 days)
  4. Reservation status is updated to FULFILLED
  5. reservation.fulfilled event is published

8.3 Reservation Expiration (Scheduled Task)

Executed every 30 minutes:

  1. Query reservations where status=PENDING and expires_at < current time
  2. Mark status as EXPIRED
  3. Release stock (available_stock +1)
  4. reservation.expired event is published

8.4 Reservation Inquiry

Users can view:

  • Active reservation list: book title, reservation time
  • Remaining pickup time (countdown)

9. Dictionary Lookup & Vocabulary Book (vocabulary.feature)

9.1 Dictionary Lookup

Lookup Entry Points:

  • Tap a word in the text during listening-reading
  • Manually input a word to look up

Lookup Results:

  • Word, phonetic transcription (IPA format), part of speech, Chinese definition, example sentence
  • Pronunciation playback button (tap to play standard pronunciation)

Lookup Data Sources:

  1. ECDICT local dictionary (3.38 million entries, primary source)
  2. Free Dictionary API (fallback when local dictionary has no match)
  3. If neither source has the word → display "Word not found, recorded, will be added as soon as possible"

Non-paid User Restrictions:

  • Only applies to status=0 (trial users)
  • Daily lookup count limit is controlled by the vocab_lookup_limit configuration
  • Observation period users and full members have unlimited lookups

9.2 Vocabulary Book

Adding to Vocabulary Book:

  • Tap "Add to Vocabulary Book" in the lookup popup
  • If the word is already in the vocabulary book → increment lookup_count, do not create a duplicate entry

Vocabulary Book List:

  • Displays all collected vocabulary: word, Chinese definition, date added, source book
  • Supports filtering by status: Learning / Mastered

Review Mode:

  • Displays the English word → user taps to view the Chinese definition
  • User can select "Mastered" or "Continue Learning"
  • Once marked as "Mastered", the word is moved to the "Mastered" list and no longer appears in the review list

Vocabulary Statistics:

  • Total vocabulary count, number currently learning, number mastered
  • Monthly new vocabulary trend

Vocabulary Highlighting:

  • When a user reads content containing previously recorded vocabulary, the word is highlighted in the text
  • Tapping the highlighted word displays its definition again

11. Advancement System (advancement.feature)

11.1 Level Management

Level Definition:

  • A–Z, 26 levels in total, each with a name, badge emoji, and sort order
  • Each level requires completing 5 books by default (configurable per level)
  • Minimum quiz pass rate defaults to 80% (configurable)
  • Each level has a maximum readable AR level limit

Initial Level Assignment:

  • When a child first registers, they are automatically assigned the lowest level
  • The current level badge is displayed

Level Information Display:

  • Current level name and badge
  • Advancement requirements: number of books to complete, quiz pass rate
  • Current progress: books completed, quizzes passed

11.2 Reading Submission

Submission Flow:

  1. Child completes listening-reading (finishes the last page) → a ReadingSubmission record is automatically created
  2. Submission status is "Pending Review" (STATUS_PENDING)
  3. Teacher receives a pending review notification

Teacher Review:

  • Approved → submission status changes to "Approved" (STATUS_APPROVED) → books read at this level +1
  • Rejected → submission status changes to "Rejected" (STATUS_REJECTED) → child receives a rejection notification

11.3 Quiz

Question Bank Management:

  • Each book requires at least 5 multiple-choice questions (A/B/C/D, choose one)
  • Each question has a correct answer, explanation, and difficulty level (1–5)
  • Teachers are responsible for creating and maintaining the question bank

Quiz Creation Flow:

  1. Teacher creates a quiz for a child → selects questions from the question bank
  2. A Quiz instance is created: status=IN_PROGRESS, total_questions=number of questions

Quiz-Taking Flow:

  1. Child answers questions one by one
  2. Submits answers → system scores automatically
  3. Quiz status changes to COMPLETED, correct_count and score are recorded

11.4 Quiz Results

Passing Criteria:

  • Accuracy >= 80% (configurable) → quiz passed
  • Upon passing: the borrowing record's assessment status is updated to PASSED

Failing:

  • Accuracy < 80% → quiz not passed
  • The borrowing record's assessment status remains PENDING
  • A "You may retake the quiz" prompt is displayed

Points Credit:

  • After passing the quiz, the book's word_count is credited to the child's points (child.total_words_read)
  • Points Deduplication: The same child+book combination only counts word_count once; retaking the quiz does not double-count points

11.5 Level Advancement

Advancement Criteria (all conditions must be met):

  1. Sufficient books read at current level (books_read_at_level >= required_books)
  2. At least 1 quiz passed at current level (quizzes_passed_at_level >= 1)

Advancement Flow:

  1. System checks advancement criteria → conditions met
  2. Current level is marked as non-current (is_current=False)
  3. A new level record is created (is_current=True)
  4. level.advanced event is published → advancement certificate is auto-generated
  5. New level badge is awarded
  6. Borrowing limit increases (max_borrow_count)

Highest Level:

  • If already at the highest level and attempting to advance → display "Already at the highest level" prompt

11.6 Achievement System

Achievement Types:

  • Advancement achievements: first advancement, reaching a specific level
  • Reading milestones: completing 10/50/100 books
  • Consecutive check-in: 7/21/100 consecutive check-in days
  • Perfect quiz score: achieving 100% accuracy

Achievement Display:

  • The achievements page displays all earned badges (in color)
  • Unearned badges are displayed in grayscale
  • Leaderboard: ranked by reading volume in descending order, top 3 receive gold/silver/bronze medals

12. Level Certificate (level_certificate.feature)

12.1 Certificate Generation

Auto-Generation:

  • When a child advances to a new level, the system automatically generates a level-up certificate
  • The certificate includes: child's name, English name, level name, badge emoji, certificate number (MW-YYYYMMDD-random code)
  • The certificate is rendered via an HTML template and can be exported as PDF

Idempotency:

  • Duplicate certificates are not generated for the same level; a repeated request returns the existing certificate

12.2 Certificate Query

  • Query the full list of certificates for a child
  • View details of a single certificate
  • Returns an empty list when no certificates exist

13. Observation Period Report (observation_report.feature)

13.1 Report Generation

Auto-Generation Conditions:

  • The child is an observation period member (status=OBSERVATION)
  • 30 days have elapsed since member_start_time
  • No report has been generated previously

Generated Content:

  • Observation period start and end dates
  • Number of books read (count of approved ReadingSubmission records)

  • Quiz statistics: number of attempts, number of passes
  • Level at the end of the period

No Generation Before 30 Days:

  • Child has been in the observation period for only 15 days -- no report is generated

13.2 Report Content

  • Number of books read and total word count
  • Number of quizzes passed
  • Teacher comments (teachers can add comments manually)
  • Average daily reading minutes

13.3 Report Query

  • Parents can view report details
  • Returns empty when no report exists

15. Refund Application Process (refund_application.feature)

15.1 Refund Rules

Parent-Child Class Refund:

  • Before the class starts: full refund of 99 (CNY)
  • After the class starts: no refund available

Observation Period Refund:

  • Refund amount = Amount paid - (Amount paid / 30 x Days used)
  • Example: Paid 500 (CNY), used 10 days -- Refund = 500 - 500/30x10 = 333.33 (CNY)
  • Multi-child discount: calculated based on the amount actually paid (e.g., 10% discount, paid 450 (CNY) -- 450 - 450/30x15 = 225 (CNY))

Official Member Refund:

  • Refund amount = Amount paid - (Amount paid / 365 x Days used)
  • Example: Paid 5400 (CNY), used 30 days -- Refund = 5400 - 5400/365x30 = 4956.16 (CNY)
  • Multi-child discount: calculated based on the amount actually paid

General Rules:

  • Refund amounts are calculated by the server; client-side values are not trusted
  • Refunds are processed within 3-7 business days and returned via the original payment method
  • Refund guard check: before a refund application is accepted, the system verifies that the child has no active borrowing records (BORROWING/OVERDUE); otherwise, the refund is rejected

15.2 Refund Review

Review Process:

  1. User submits a refund application
  2. Admin reviews the application (approve / reject)
  3. If approved -- the system automatically initiates the refund; the user receives a "Refund Approved" notification
  4. If rejected -- the user receives a "Refund Rejected" notification, including the rejection reason

15.3 Membership Expiry Reminder (Scheduled Task)

Reminder Schedule (configurable):

  • 30 / 15 / 7 / 3 / 2 / 1 day(s) before membership expiry and on the expiry day -- sends a WeChat subscription message reminder
  • Notification priority: "Medium"
  • Pushed to the user's WeChat

16. Voice Reading and Practice (voice_reading.feature)

16.1 Voice Recording

Recording Flow:

  1. The child selects a passage of English text
  2. Taps the "Read Aloud" button
  3. Reads the text aloud
  4. The system records the audio
  5. The recording is saved to the cloud (audio_url)

Playback Feature:

  • Tap the "Playback" button to play back the recording just made
  • The reading text is displayed simultaneously for comparison

16.2 Reading Records

  • Displays a list of all voice reading records
  • Each record shows: book title, reading text, recording duration, date

16.3 Voice Reading and Check-in Integration

  • Upon completing a voice reading, the system automatically triggers a reading check-in (if the child has not checked in today)

16.4 Speech Evaluation (Future Version)

Pronunciation Scoring (planned):

  • The system returns a pronunciation score (0-100)
  • Sub-scores: pronunciation accuracy, fluency, completeness
  • If the score is below 60: displays pronunciation improvement suggestions and highlights words with inaccurate pronunciation
