## 测试运行结果

| 测试套件 | 用例数 | 通过 | 失败/报错 | 通过率 | 备注 |
|----------|--------|------|-----------|--------|------|
| `pytest tests/unit/` | 100 | 94 | 6 | 94.0% | 源码返回类型错误（Service 调用 `.model_dump()`） |
| `behave features/` | 137 场景 | 134 | 4 报错 | 97.8% | 同源返回类型错误 |
| `scripts/formal_test_v2.py` | 119 | 101 | 18 | 84.9% | 测试脚本管理端路径过期（`/admin/*` vs `/admin/api/*`） |

> **结论**：当前代码无法直接“测试全绿”。测试失败主要源于两类问题：
> 1. **源码返回类型不一致**（6 单元 + 4 BDD 失败）
> 2. **集成测试脚本路径过期**（18 个假阴性失败）
> 修复后预计可恢复到 98% 以上通过率。

## 各维度关键问题摘要

### 维度 1：需求覆盖度（P0×5 / P1×9 / P2×8 / P3×2）

- **P0**：阅读提交未自动创建、测验误增已读书数、报名前置条件缺失、权益转让目标状态校验不足、亲子课退款规则缺失。
- **P1**：预约借书缺押金校验、收费活动取消未退款、查词无 Free Dictionary API 兜底、观察期支付后未分配老师、会员到期未转 EXPIRED、借阅允许 REFUNDING 押金、借书未校验 BookCopy 状态、预约取消未释放库存、逾期后音频锁定未实现、学习报告缺语音朗读统计。
- **P2/P3**：多孩优惠逻辑偏差、活动提醒优先级、书架容量限制、查词限制基于 user_id、观察期报告缺日均分钟、朗读未自动打卡、语音评测字段超前、收藏夹过度开发。

### 维度 2：API 接口（P0×2 / P1×4 / P2×3 / P3×1）

- **P0**：`/admin/api/oplogs` 未认证；用户端 Schema 未统一 `extra="forbid"`。
- **P1**：管理端多个关键端点使用裸 `dict` 入参；`RefundResponse` 金额字段用 `float`；活动批量签到未限定管理员角色；会员升级接口未校验孩子归属（IDOR 风险）。
- **P2/P3**：多个列表端点未声明 `response_model`；`OrderCreate.type` 用整数范围而非枚举；`BatchCheckinRequest` 未限制最大长度；配置 PUT 接口用 query 传 value。

### 维度 3：业务逻辑（P0×4 / P1×13 / P2×3）

- **P0**：多孩优惠错误、观察期/正式会员前置条件缺失、测验积分去重错误、阅读提交审核不触发晋级统计。
- **P1**：押金状态机缺 REFUNDED 终态、丢书罚款未扣押金、预约取书未绑定 BookCopy、用户取消预约未释放库存、活动取消并发下溢风险、晋级条件默认要求 5 次测验、活动组织者取消后自动退款无执行链路、逾期罚款与 outstanding_fines 双写分叉、续费折扣与多孩优惠叠加规则不明。
- **P2**：观察期报告生成 stub、退款与借书竞态、微信退款失败无审计。

### 维度 4：数据库与模型（P0×2 / P1×3 / P2×4 / P3×1）

- **P0**：关键业务表缺少唯一约束（借阅、Quiz、成就、阅读提交）；多处高频字段缺少索引。
- **P1**：`extend_existing=True` 滥用；软删除过滤不一致；库存字段缺乏数据库级一致性约束；`Child.outstanding_fines` 是缓存字段无重建机制。
- **P2/P3**：`Child.current_level_id` 冗余与 `ChildLevel.is_current` 可能不一致；状态字段裸整数无 CHECK；时间字段未建索引；部分文本字段长度不足。

### 维度 5：安全审查（P0×0 / P1×5 / P2×2 / P3×2）

- **P1**：Token 无黑名单/吊销机制；管理员代客退款绕过未还书/罚款校验；订单状态可直接标记为已支付且无审计；分片上传路径遍历；`/admin/api/oplogs` 未认证写入。
- **P2/P3**：限流器为进程内存；文件上传仅校验扩展名；导出包含 openid/手机号；数据导入脚本 SQL 拼接；DEBUG 硬编码测试 Token；CORS 缺 PATCH。

### 维度 6：错误处理（P0×0 / P1×1 / P2×3 / P3×1）

- **P1**：数据库唯一/外键冲突未友好处理，直接返回 500。
- **P2/P3**：微信支付回调在证书未配置时 500；微信退款异步失败仅写站内消息；外部词典 API 异常静默返回 None；缺少集中式日志配置。

### 维度 7：性能审查（P0×0 / P1×4 / P2×4 / P3×2）

- **P1**：活动报名列表 N+1；全表 OFFSET 分页；`check_due_date_reminders` 循环全表扫描；`ConfigService` 进程缓存无法跨 worker。
- **P2/P3**：月报串行生成全量会员；`get_upgrade_options` 无联合索引；搜索未用全文索引；同步 SQLAlchemy 与 `async def` 混用；健康检查缺依赖探测；定时任务无监控。

### 维度 8：代码质量（P0×0 / P1×1 / P2×4 / P3×2）

- **P1**：系统管理路由存在大量跨层直接查库。
- **P2/P3**：`BookService` 重复 `update_book`；多处路由用裸 `data: dict`；`AdminService.update_admin` 角色比较低效；返回格式不一致；`BaseSchema` 未设 `extra="forbid"`；回收站模块硬编码；`User.password` 字段未使用。

### 维度 9：测试覆盖（P0×1 / P1×5 / P2×4 / P3×1）

- **P0**：Service 方法返回类型与签名不一致，导致 6 单元 + 4 BDD 失败。
- **P1**：大量 Service 无单元测试；缺少并发、API 集成、事务回滚测试。
- **P2/P3**：Feature 数量与文档不符（137 vs 138）；`formal_test_v2.py` 路径过期；边界值/异常路径不足；测试用 SQLite 与生产 MySQL 有方言差异；测试命名不统一。

### 维度 10：配置与部署（P0×0 / P1×2 / P2×4 / P3×2）

- **P1**：缺少锁定版本的依赖清单；Alembic 模型与数据库存在漂移。
- **P2/P3**：`alembic.ini` 硬编码数据库连接；无集中日志与告警；缺少数据迁移与回滚方案文档；生产环境默认数据库密码为空；定时任务无分布式锁；健康检查暴露版本号。

### 维度 11：前端管理页面（P0×1 / P1×3 / P2×6 / P3×2）

- **P0**：`admin.js` 全局拦截所有表单提交，未带 Authorization 头且使用 FormData，导致管理端 CRUD 无法使用。
- **P1**：`content.html` 未接入后端；`operation_logs.html` 和 `recycle_bin.html` 端点前缀错误。
- **P2/P3**：登录页调试 alert；CSS 大量硬编码颜色；图书上传绕过统一 api 客户端；`content.html` title block 污染；分页器重复实现；仪表盘静态占位数据；大量内联样式。

### 维度 12：PRD 与 Feature（P0×0 / P1×0 / P2×7 / P3×2）

- **P2**：借阅/预约 Feature 状态名与 PRD 不一致；缺少关键负面场景；多孩优惠未覆盖观察期触发条件；季度/半年会员边界规则描述不完整；EXPIRED 可购买类型未明确；阅读提交自动创建未覆盖。
- **P3**：PRD 缺少版本变更日志；语音评测 Feature 与规划阶段不一致。

## 总体结论

### 整体评级：C（有条件通过）

MegaWords V3.5 项目主体框架已成型，22 个业务域、管理后台 33 个页面、测试框架均已落地。然而，**12 个 P0 致命问题 + 42 个 P1 严重问题**表明当前代码距离可上线还有明显差距。主要风险集中在：

1. **业务规则失效**：晋级、报名、退款、库存、押金等核心流程存在规则缺失或实现偏差。
2. **数据一致性风险**：关键表缺少唯一约束与索引，高并发下易产生脏数据。
3. **安全与部署基线不足**：Token 无法吊销、未授权接口可写日志、缺少锁定版本依赖与 Alembic 漂移。
4. **前端管理端不可用**：`admin.js` 全局表单拦截导致核心 CRUD 功能无法使用。
5. **测试无法全绿**：源码返回类型错误与测试脚本路径过期导致当前测试失败。

### 关键风险项 Top 10

1. 阅读提交未自动创建，晋级与报告数据链断裂（P0）
2. 测验通过误增已读书数，晋级条件失真（P0）
3. 多孩优惠逻辑错误，直接资损与客诉（P0）
4. 观察期/正式会员报名前置条件缺失，商业 funnel 失效（P0）
5. 关键表缺少唯一约束，高并发数据污染（P0）
6. `/admin/api/oplogs` 未认证，安全漏洞（P0）
7. `admin.js` 全局表单拦截，管理端 CRUD 不可用（P0）
8. 缺少锁定版本依赖清单与 Alembic 漂移，部署不可预测（P1）
9. Token 无黑名单/吊销，stolen token 可长期滥用（P1）
10. 管理员资金操作绕过校验且无审计日志（P1）

### 建议修复优先级

#### 第一优先级（1-2 周内，P0 + 关键 P1）
- 修复 12 个 P0 致命问题。
- 修复 Service 返回类型错误，恢复 10 个测试（6 单元 + 4 BDD）。
- 移除 `admin.js` 全局表单拦截，修正操作日志/回收站端点，实现 `content.html` 后端对接。
- 补充关键表唯一约束与高频索引（Alembic 迁移）。
- 补充锁定版本依赖清单，修复 Alembic 模型漂移。
- 修复 `/admin/api/oplogs` 未认证、分片上传路径遍历。
- 统一管理员资金操作校验与审计日志。

#### 第二优先级（3-4 周内，P1）
- 押金状态机补全 REFUNDED 终态、丢书罚款扣押金。
- 预约取书绑定 BookCopy、取消预约释放库存。
- 逾期音频锁定、学习报告语音朗读统计。
- 查词 Free Dictionary API 兜底、观察期支付后分配老师。
- 会员到期状态转换、活动取消退款与并发控制。
- Token 黑名单/吊销机制、限流 Redis 化。
- 管理端裸 `dict` 入参全面替换为 Pydantic Schema。

#### 第三优先级（上线后 1 个月内，P2/P3）
- 性能优化（N+1、分页、缓存、索引）。
- 可观测性增强（日志、告警、健康检查、定时任务监控）。
- Feature 文件状态术语统一、负面场景补充、PRD 边界规则完善。
- 代码质量重构（跨层调用、重复代码、Schema 统一）。
