## 关键问题 1：P0-3 报名前置条件

**问题**：亲子课完成和测评报告是否必须系统内有记录？建议改为管理员可手动标记。

**专家回复**：

- **接受降级为 P1**。但必须在本次修复中实现"管理员可手动标记已完成亲子课/测评"的能力，否则系统无法闭环。
- **原因**：
  - 当前代码里 `Assessment` 模型存在，但与亲子课无直接关联，且线下完成的场景确实需要人工确认。
  - 如果硬校验系统内记录，会真实卡住线下已完成的家庭，这比"不校验前置条件"更糟。
- **建议实现方案**：
  1. 在 `Child` 或 `Assessment` 表中增加 `parent_course_completed`（布尔）和 `assessment_completed`（布尔）字段，由管理员/教务老师在后台标记。
  2. 观察期/正式会员订单创建时，校验 `parent_course_completed AND assessment_completed`。
  3. 不强制要求系统内有 `Assessment` 记录本身，但要求完成标记存在。
  4. 前端管理端增加"标记已完成亲子课"和"标记已完成测评"的入口。
- **判定**：P0-3 和 P0-7 合并为 **P1**，条件是在本次修复中完成"手动标记 + 校验"的闭环。

## 关键问题 2：P0-6 多孩优惠叠加

**问题**：续费折扣与多孩折扣是否可叠加？

**专家回复**：

- **不可叠加**。多孩优惠与续费折扣是两种独立的优惠策略，叠加会导致资损和规则混乱。
- **建议规则**：
  - 同一张订单只能适用一种折扣。
  - 优先级：多孩优惠（0.9）与续费优惠（0.9）取 **最低折扣**（即 0.9），但不可连续乘算（不可变成 0.81）。
  - 如果未来有优惠券体系，也按"取最低折扣"原则，避免乘法叠加。
- **当前实现的修复**：
  - `_apply_discount` 的逻辑应改为：检查同一 user 下是否存在 `status in (OBSERVATION, OFFICIAL)` 的孩子，存在则第 2 孩起 0.9 折。
  - 续费场景（缓冲期内）的 0.9 折与多孩 0.9 折互斥，取 0.9。
- **判定**：P0-6 仍为 **P0**（直接资损），修复时同时明确折扣互斥规则。

## 关键问题 3：P0-10 / P0-11 降级

**问题**：唯一约束和索引是否接受降级为 P1（Sprint 2 修复）？

**专家回复**：

- **接受降级为 P1**，但 Sprint 1 必须加应用层防御，不能完全放任。
- **原因**：
  - 当前测试环境数据量小，儿童阅读平台不是秒杀场景，并发风险确实不高。
  - 但关键表（借阅记录、Quiz、成就、阅读提交）的重复数据会导致孩子晋级、积分、奖品数据永久错误，属于数据质量问题，不能无限期拖后。
- **Sprint 1 应用层防御**：
  - 在 `BorrowRecord.create`、`Quiz.create`、`Achievement.grant`、`ReadingSubmission.create` 等关键写操作前，使用数据库唯一键或 `SELECT FOR UPDATE` 或 `INSERT ... ON DUPLICATE KEY` 模式做幂等保护。
  - 尤其是活动报名、借书、还书、退款等资金/库存敏感操作，必须加锁或唯一约束。
- **Sprint 2 数据库层补齐**：
  - 用 Alembic 迁移为以下场景补充唯一约束（示例）：
    - `(child_id, book_id, status in (PROGRESS, BORROWED))` 防止同一孩子重复借阅同一本书未还。
    - `(child_id, book_id, passed=True)` 防止同一本书重复通过测验并重复计分。
    - `(child_id, achievement_type_id)` 防止重复发同一成就。
- **判定**：P0-10 和 P0-11 降为 **P1**，但 Sprint 1 必须完成应用层锁，Sprint 2 必须完成数据库约束/索引。

## 关键问题 4：P0-13 admin.js 全局表单拦截

**问题**：是否有页面依赖全局拦截？修复策略是移除还是改为带 auth 的 JSON 提交？

**专家回复**：

- **结论：直接移除全局拦截逻辑**。各页面已经自行使用 `api.post/put/delete` 处理表单提交，全局拦截属于重复且错误的代码。
- **验证**：
  - 从管理页面代码看，图书（booklist/books）、老师（teachers）、场馆（venues）、级别（levels）、成就（achievements）等页面都使用 `submitXxx()` 函数调用 `api.*`。
  - 没有证据表明任何页面依赖这个全局拦截器。即使有极少数页面使用原生 `<form>`，也应该改成调用 `api.*` 而非依赖全局拦截。
- **修复策略**：
  1. 删除 `backend/static/admin/js/admin.js:345-364` 的 `DOMContentLoaded` 表单提交监听器。
  2. 检查所有模板中是否还有 `<form>` 原生提交，如有，改为调用 `api.post/put` + JSON body。
  3. 如需统一防重提交，提供 `submitWithLock(fn)` 工具函数，不覆盖默认行为。
  4. 修复 `operation_logs.html` 和 `recycle_bin.html` 的端点前缀为 `/admin/api/*`。
  5. 实现 `content.html` 的内容管理后端对接。
- **判定**：P0-13 仍为 **P0**，管理端 CRUD 不可用是致命缺陷。

## 关键问题 5：朗读自动打卡

**问题**：PRD 16.3 要求朗读完成自动打卡，是否需要在本次修复中实现？

**专家回复**：

- **不需要在本次 P0 修复冲刺中实现**。建议作为独立 P1 或 P2 功能项，在下一阶段处理。
- **原因**：
  - 当前已经存在手动打卡入口（`/reading/checkin`），核心打卡功能未断裂。
  - 自动打卡是"体验优化"，不是"流程阻塞"。
  - 本次冲刺应聚焦 P0/P1 级别的数据错误、资金风险、功能不可用问题。
- **建议实现方式**（后续阶段）：
  - 在 `ReadingSubmission` 或 `AudioRecording` 状态变为审核通过时，自动调用 `ReadingService.check_in()`。
  - 或定义一个事件 `ReadingCompleted`，由事件处理器触发打卡，保持与测验通过事件一致的架构风格。
- **判定**：本次不纳入修复范围，标记为 **P2（体验优化）**，下一阶段实现。
