## 项目背景

MegaWords 是一个儿童英文阅读管理平台，采用 OMO 模型（线下实体书借阅 + 线上音频伴读）。

- **后端**：Python 3.13 + FastAPI + SQLAlchemy 2.0 + Pydantic V2
- **数据库**：MySQL 8.0（utf8mb4），45 张表
- **管理端**：PC Web 后台，33 个页面（含 base.html）
- **API 数量**：约 180 个端点
- **领域模块**：22 个
- **定时任务**：14 个（APScheduler）
- **测试**：100 单元测试（pytest）+ 138 BDD 场景（behave）

**输出格式**：

### 维度 2：API 接口审查

**目标**：验证所有 API 端点的规范性、一致性、安全性。

**检查项**：
1. **路由规范**：
   - URL 命名是否遵循 RESTful 规范（名词复数、层级合理）
   - HTTP 方法是否正确（GET 查询、POST 创建、PUT/PATCH 更新、DELETE 删除）
   - 是否存在路由冲突或重复注册
   - 前缀是否统一（`/admin/api/` vs `/api/` vs 其他）

2. **请求/响应规范**：
   - 所有端点是否声明了 `response_model`
   - 请求参数是否使用 Pydantic Schema（而非裸 `dict`）
   - 错误响应格式是否统一（`{"detail": "..."}` 或自定义格式）
   - 分页接口是否统一格式（`{items, total, page, page_size, has_next}`）

3. **安全审查**：
   - 所有管理端接口是否都有认证（`get_current_admin` 或 `require_role`）
   - 是否存在未授权可访问的敏感接口
   - 角色权限是否正确（ADMIN vs STAFF vs TEACHER）
   - 是否存在 IDOR（越权访问其他用户数据）风险

4. **输入校验**：
   - Pydantic Schema 是否设置了 `extra="forbid"`（拒绝未知字段）
   - 字段长度限制是否合理（`max_length`）
   - 数值范围限制是否合理（`ge`, `le`）
   - 必填字段是否正确标记

**输出格式**：
```
| 端点 | 方法 | 认证 | 角色 | Schema | 问题 | 严重度 |
|------|------|------|------|--------|------|--------|
```

### 维度 3：业务逻辑审查

**目标**：验证核心业务流程的正确性和边界处理。

**重点审查模块**：

1. **借阅流程**：
   - 借书 → 创建 BorrowRecord → 更新 BookCopy 状态 → 生成"正在阅读"
   - 还书 → 更新 BorrowRecord → 释放 BookCopy → 计算逾期罚款
   - 逾期 → 锁定音频播放 → 生成罚款单
   - 丢失 → 定价 × 1.5 倍罚款

2. **押金系统**：
   - 状态机：UNPAID → PAID → REFUNDED / DDEDUCTED
   - 退款校验：无未还书 AND 无未缴罚款
   - 并发安全：押金退款与借书操作的竞态条件

3. **预约取书**：
   - 线上预约 → 锁定 `offline_available` 库存
   - 72 小时过期 → 自动释放库存
   - 到店取书 → 转为正式借阅

4. **活动报名**：
   - 并发控制：SQL 原子更新 `WHERE current_participants < max_participants`
   - 取消规则：活动开始前 N 小时内不可取消

5. **测评系统**：
   - 通过率阈值：≥80%（5 题答对 4 题）
   - 积分计算：同一 child + book 只计一次
   - 防刷分机制

6. **订单系统**：
   - 订单类型：亲子课 / 观察期 / 正式会员 / 季度 / 半年
   - 多孩优惠：第 2 孩起打 9 折
   - 未支付自动关闭：30 分钟

**检查项**：
- 每个流程的正常路径是否完整
- 异常路径是否处理（余额不足、库存为 0、重复操作等）
- 事务边界是否正确（哪些操作需要在同一事务中）
- 并发场景是否安全（竞态条件、死锁风险）

### 维度 4：数据库与模型审查

**目标**：验证数据模型设计的合理性和一致性。

**检查项**：

1. **表结构**：
   - 所有表是否继承 `BaseModel`（含 id, create_time, update_time, is_deleted）
   - 外键关系是否正确声明
   - 索引是否合理（高频查询字段、联合索引）
   - 字段类型是否合适（金额用 Decimal/整数分，不用 float）

2. **ORM 关系**：
   - `relationship` 是否正确配置（`back_populates`, `foreign_keys`）
   - 是否存在循环依赖或延迟加载问题
   - `extend_existing = True` 是否必要（多文件定义同一表时）

3. **软删除**：
   - 所有查询是否都带 `is_deleted == 0` 过滤
   - 软删除后关联数据如何处理（是否需要级联软删除）
   - 回收站恢复后关联数据是否完整

4. **数据一致性**：
   - 库存字段（total_stock, available_stock, offline_available）是否保持一致
   - 计数字段（current_participants）是否与实际记录数一致
   - 状态字段是否与关联表状态一致

**输出格式**：
```
| 表名 | 问题 | 影响范围 | 修复建议 | 严重度 |
|------|------|---------|---------|--------|
```

### 维度 5：安全审查

**目标**：识别潜在的安全漏洞。

**检查项**：

1. **认证与授权**：
   - JWT Token 是否有过期机制
   - Token 黑名单 / 吊销机制是否存在
   - 密码哈希算法是否安全（bcrypt）
   - 登录暴力破解防护是否有效

2. **输入注入**：
   - SQL 注入：是否全部使用参数化查询（SQLAlchemy ORM 默认安全，但需检查原生 SQL）
   - XSS：管理端 HTML 模板是否对用户输入做了转义
   - 命令注入：是否有 `os.system` / `subprocess` 调用

3. **敏感数据**：
   - 日志中是否打印了密码、Token、手机号等敏感信息
   - API 响应中是否泄露了内部 ID、堆栈信息
   - `.env` 文件是否在 `.gitignore` 中

4. **业务安全**：
   - 资金操作是否有幂等性保护（重复退款、重复支付）
   - 关键操作是否有审计日志
   - 文件上传是否有类型/大小限制

---

### 维度 6：错误处理审查

**目标**：验证所有异常场景的处理是否符合"零 500"要求。

**检查项**：

1. **全局异常处理**：
   - `main.py` 是否注册了全局 Exception handler
   - `BusinessException` 是否返回友好 JSON
   - 未捕获异常是否返回 500 + 友好消息（而非堆栈）

2. **业务异常**：
   - 是否定义了完整的异常层级（ValidationError, NotFoundError, ConflictError, ForbiddenError, UnauthorizedError, RateLimitError）
   - 每个异常是否返回了合适的 HTTP 状态码
   - 错误消息是否对用户友好（中英文、不暴露内部细节）

3. **数据库异常**：
   - 唯一约束冲突是否捕获并返回友好消息
   - 外键约束冲突是否处理
   - 连接超时/断开是否有重试机制

4. **外部服务异常**：
   - 微信 API 调用失败是否处理
   - 短信发送失败是否处理
   - 超时设置是否合理

**输出格式**：
```
| 异常场景 | 当前行为 | 期望行为 | 文件位置 | 严重度 |
|----------|---------|---------|---------|--------|
```

### 维度 7：性能审查

**目标**：识别潜在的性能瓶颈。

**检查项**：

1. **N+1 查询**：
   - 列表接口是否使用了 `joinedload` / `selectinload` 避免 N+1
   - 嵌套序列化是否触发了延迟加载

2. **分页效率**：
   - 大表分页是否使用了游标分页（而非 OFFSET）
   - COUNT 查询是否可以优化（缓存或近似值）

3. **索引使用**：
   - 高频查询字段是否有索引
   - 联合查询是否使用了联合索引
   - EXPLAIN 分析是否覆盖了主要查询路径

4. **缓存策略**：
   - 系统配置是否缓存（ConfigService 带 TTL）
   - 静态数据（级别、成就）是否缓存
   - 用户信息是否在请求间复用

## 审查原则

1. **零容忍**：不放过任何 500 错误、安全漏洞、数据不一致
2. **证据驱动**：每个问题必须指出具体文件和行号
3. **业务导向**：从用户（培训运营人员）视角验证功能
4. **企业标准**：按 SOC 2 / ISO 27001 等标准评估安全和合规
5. **可操作性**：每个问题都附带具体的修复建议
